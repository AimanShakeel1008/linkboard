Why this phase exists

For three phases we built everything by hand. We opened raw sockets, parsed HTTP requests character by character, assembled responses byte by byte, and routed with a giant chain of if/elif. That was the whole point — to prove, with our own hands, that the web is not magic. Now we cash in that understanding.

This phase does two big things at once. First, we stop hand-writing the plumbing and adopt a real web framework (FastAPI), because now that we know exactly what routing, parsing, and serialization are, letting a tool do them is a superpower, not a mystery. Second, we redesign LinkBoard as a proper API — a clean, predictable interface that other programs (and, in Phase 5, our own web frontend) can talk to. By the end, LinkBoard v1 will let you submit links, list them, upvote, and comment — all as real HTTP calls returning JSON.

And then, right at the moment it feels like a real app, we break it on purpose. Two people upvote the same link at the same instant — and one of the votes vanishes. That disappearing vote is not a bug in our logic; it's the fundamental limit of storing data in a plain file, and feeling it here is what will make Phase 6's database — and the word "transaction" — feel inevitable rather than academic.

Part A — What is an API? Ordering from a menu

The term API (Application Programming Interface) sounds forbidding, but the idea is something you use every day without thinking. When you sit down at a restaurant, you don't walk into the kitchen, find the chef, and explain how to sear a steak. You read a menu, say "the number 4, please," and a while later the dish arrives. The menu is a small, stable, public list of things you're allowed to ask for. The kitchen — the pots, the recipes, the chaos — stays hidden behind it.

An API is exactly that: a menu one program offers to other programs. It's a defined set of requests you're allowed to make ("list the links," "submit this link," "upvote link 3") and the responses you'll get back — with all the messy internals (files, databases, algorithms) hidden behind it. You order by name; you don't need to know how the kitchen works, and the kitchen is free to change how it cooks as long as the menu stays the same.

LinkBoard's API is a web API specifically: the "menu items" are HTTP requests, and the "dishes" are JSON responses. Everything you learned in Phase 3 — methods, paths, status codes, headers — is the vocabulary this menu is written in. An API is just a disciplined, well-labelled way of using HTTP.

APIs are what let software be built out of software. Your weather app doesn't measure the weather — it calls a weather API. A checkout page doesn't handle your card — it calls Stripe's API. Sign in "with Google" is Google's API. Entire companies exist purely as an API with no user-facing app at all (Twilio for text messages, Stripe for payments). "Design the API" is often the first real task in a system-design interview, because the API is the contract everything else is built around — get it right and the rest has a shape; get it wrong and every client suffers. You're about to design one for LinkBoard.

Part B — Why frameworks exist: the plumbing, now automatic

Let's be honest about what our hand-written server cost us. To add a single new route in Phase 3, we had to: add a branch to the if/elif tree, manually read the body bytes, hand-parse whatever format they were in, check every field ourselves, build a Response object, and format it to bytes. Every route repeated most of that. It was educational — and it was a lot of fiddly, repetitive, bug-prone work.

A web framework is a library that does all that repetitive plumbing for you, so you write only the part that's actually unique to your app. We're using FastAPI, a modern Python framework. Here is precisely what it takes off our hands — and you'll recognise every item, because we built each one by hand:

  • Routing — instead of an if/elif tree, you tag a function with @app.get("/links") and the framework calls it when that method+path arrives.
  • Parsing — the framework reads the request body and turns the JSON into a Python object; you never touch raw bytes.
  • Validation — you declare the shape you expect ("a title string and a url string"); a request missing a field is auto-rejected with a clear error before your code runs.
  • Serialization — you return a plain Python dictionary; the framework turns it into a JSON response with the right headers.
  • Free extras — FastAPI even generates an interactive documentation page at /docs from your code, so anyone can explore the API in a browser. We never wrote a line of it.
📝 Why we didn't start with a framework

Plenty of courses would have opened at Phase 4 — "install FastAPI, here's a route." You'd have a working app and no idea what any of it meant. Because we built the socket server first, every one of those magic conveniences is now a recognition, not a mystery: "oh, @app.post is the routing I did with if path == …; the auto-JSON is the build_response encoding I did by hand." A framework you understand is a tool; a framework you don't is a cage. That's the whole reason for the order of this course.

Part C — REST: designing around resources and verbs

We could expose any jumble of endpoints we liked (/getAllTheLinks, /doAnUpvoteNow). But there's a widely-followed style for designing web APIs that makes them predictable, and it's called REST. You don't need its academic definition; you need its one core habit, which is genuinely simple:

Nouns are resources (the things), expressed as paths. Verbs are HTTP methods (the actions), from Phase 3. You combine a verb and a noun to say what you want.

A resource is a "thing" your app knows about — for LinkBoard, a link and a comment. In REST, you name resources with paths, and you act on them with the HTTP methods you already learned, rather than inventing a new verb-word for every action. So instead of /getAllTheLinks, you say GET /links — the method is the verb. Here's LinkBoard's whole API expressed that way:

You want to…Verb + nounMethodPath
list all linksread the linksGET/links
read one linkread a linkGET/links/3
submit a new linkcreate a linkPOST/links
upvote a linkact on a linkPOST/links/3/upvote
comment on a linkcreate a comment under a linkPOST/links/3/comments

Notice two elegant things. First, /links means different things with different verbs: GET /links reads the collection, POST /links adds to it. Same noun, method chooses the action — exactly the safe/idempotent distinctions from Phase 3 doing real work. Second, a comment lives underneath the link it belongs to: /links/3/comments. That nesting mirrors the real relationship ("this comment belongs to that link") right in the URL.

Quick check: why is POST /links better REST than POST /createLink, even though both work?

Because REST puts the verb in the HTTP method and keeps the path a pure noun. POST /links says "create, within the links collection" using the method as the verb — so the same noun /links naturally supports GET (list) and POST (create) without inventing new words. /createLink bakes the verb into the path, so you'd soon need /updateLink, /deleteLink, /getAllLinks… a sprawling, unpredictable menu. The REST style means a client can guess your API: once they know links are at /links, they can reasonably expect GET /links/3 and DELETE /links/3. Predictability is the whole win.

Part D — JSON & serialization: turning objects into bytes

Our API sends and receives data as JSON. To understand why, first meet the problem it solves, which has a name worth knowing: serialization.

Inside your running program, a link is a rich live object — a Python dictionary sitting in memory with fields and values, connected to other objects, existing as a pattern of bits at some memory address. You cannot send that across a network. A socket (Phase 2) carries only a flat stream of bytes. So before an object can travel, it must be flattened into a plain sequence of bytes that the other side can rebuild. Flattening an object into bytes is serializing it; rebuilding the object from those bytes on the far end is deserializing. A serialization format is the agreed rules for how that flattening is done — and it's the very same "text is numbers plus an agreement" idea from Phase 1, scaled up from single characters to whole objects.

What JSON looks like

JSON (JavaScript Object Notation) is the most common serialization format on the web, and it's deliberately simple — human-readable text built from a handful of pieces you already understand: objects ({ } with "name": value pairs), arrays ([ ] lists), strings, numbers, true/false, and null. Here's one LinkBoard link as JSON:

{
  "id": 3,
  "title": "HTTP by hand",
  "url": "https://example.com/http",
  "points": 5,
  "comments": []
}

That's it — no surprises. It maps almost one-to-one onto a Python dictionary or a JavaScript object, which is exactly why it took over the web: the same shape works naturally in nearly every language.

JSON vs XML vs Protobuf

JSON isn't the only way to serialize. Two others come up constantly, and knowing the tradeoff is a real interview and design topic:

FormatLooks likeStrengthsWeaknesses
JSONtext: {"points":5}human-readable, universal, simple; native to browsersbigger than binary; parsing text costs CPU; no built-in schema
XMLtext: <points>5</points>very expressive, mature tooling, strong schema/validationverbose and heavy; fiddly to read and write; largely superseded by JSON for web APIs
Protobufbinary (not human-readable)tiny and very fast; strict schema; great for service-to-service at scaleneeds a schema + code generation; you can't just eyeball it; overkill for simple apps

The through-line: text formats (JSON, XML) are readable but bulky; binary formats (Protobuf) are compact and fast but opaque. The web chose JSON as its default because readability and universality usually beat raw efficiency for public APIs — you can debug a JSON response by just looking at it. When two internal services exchange millions of messages a second and nobody needs to read them by eye, Protobuf's compactness wins. LinkBoard uses JSON: it's public-facing, and clarity beats bytes here. (You'll meet Protobuf again if we reach high-throughput internal services.)

A classic design question: "your mobile app is slow on a bad connection — would switching from JSON to Protobuf help?" A thoughtful answer uses this whole section: maybe — Protobuf payloads are smaller, so on a bandwidth-starved link (Phase 2's bandwidth vs latency!) less data means faster transfers, and parsing is cheaper on a weak CPU. But you'd trade away human-readable debugging and add a schema/build step, and if the real bottleneck is latency (too many round-trips) rather than payload size, a smaller format won't help at all — you'd batch requests instead. Naming the tradeoff and questioning whether it addresses the actual bottleneck is exactly the muscle these phases build.

Part E — The data structures underneath it all

Time to look one level down. Every framework, cache, router, and database in this course is built on a small set of data structures — the basic ways a program arranges data in memory. Two of them, plus one way of measuring them, will follow you through every remaining phase. We meet them here because LinkBoard's routing and lookups already use them.

Big-O: "what happens at 1000×?"

First, the measuring stick. When we ask whether some operation is "fast," the question that actually matters for a growing system is not "how long does it take now?" but "how does the time grow as the data grows?" A method that's fine for 100 links might melt for 100 million. Big-O notation is the shorthand for that growth rate. Read it as "if the data gets 1000× bigger, what happens to the work?"

  • O(1) — constant. Same work regardless of size. Looking up a value by key in a hash map (below). The gold standard.
  • O(log n) — logarithmic. Grows painfully slowly. Finding a row via a database index (Phase 6's B-trees). 1000× the data costs only about 10× the work.
  • O(n) — linear. Work grows in step with data. Scanning every link to find one. Fine for 100, a problem for 100 million.
  • O(n²) — quadratic. Work grows with the square. A loop inside a loop over the same data. Deadly: this is what "it worked in testing and died in production" often looks like.

Arrays vs linked lists: two ways to hold a sequence

When you have a list of things (like LinkBoard's links), there are two fundamental ways to lay them out in memory, and they trade off differently:

  • An array stores the items in one continuous block, back to back. Because they're contiguous, jumping to "item number 500" is instant math (start + 500 × size) — O(1) access by position. The cost: inserting into the middle means shifting everything after it over to make room — O(n). Think of theatre seats bolted in a row: finding seat 12 is trivial; adding a seat in the middle means everyone shuffles down.
  • A linked list scatters the items anywhere in memory, and each item holds a pointer to the next. Inserting is cheap — just re-point a couple of arrows, O(1) — but finding "item number 500" means walking the chain from the start, O(n). Think of a treasure hunt where each clue points to the next: easy to splice in a new clue, but to reach the 500th you must follow all 499 before it.

Neither is "better" — they're opposite trades (fast access vs fast insertion), and choosing correctly is real engineering. Python's list (which LinkBoard uses) is an array under the hood, which is why appending is cheap but our GET /links/3 currently scans the list — O(n) — to find id 3. Fine for three links; a real database will make it fast later.

The hash map: O(1) lookup, and the soul of half this course

Now the most important data structure you'll meet all year. A hash map (also called a hash table, a dictionary, or just a "map") stores key → value pairs and can find the value for a key in O(1) — constant time, no matter how many pairs it holds. That near-magical property makes it the engine behind an astonishing amount of computing.

How does it pull off O(1)? Through a hash function — a bit of math that takes a key (say the string "links") and rapidly turns it into a number, which it uses as a direct index into an array of "buckets." Instead of searching for the key, it computes where the key must be and jumps straight there. It's the difference between scanning every page of a book for a topic (O(n)) versus using the index at the back to jump straight to the page (O(1)).

CS fundamental: the hash map is the "soul" behind a huge fraction of this course. A framework's router maps method+path → handler with one (Phase 4). A cache like Redis is essentially a giant hash map of key → value (Phase 8). A database index is a lookup structure so you don't scan every row (Phase 6). Even DNS (Phase 2) is conceptually name → address. When you see "fast lookup by key," think hash map.

Quick check: our GET /links/3 scans the list to find id 3 — what big-O is that, and what structure would make it O(1)?

Scanning the list checks links one by one until it finds id 3, so in the worst case it looks at every link — that's O(n), linear in the number of links. Storing links in a hash map keyed by id ({3: link}) instead of a plain array would turn that lookup into O(1): hash the id, jump straight to the bucket, done — the same tiny cost whether there are 3 links or 3 million. This is precisely what a database's primary-key index does for you (Phase 6). For LinkBoard's three seed links the O(n) scan is invisibly fast; the point is to see the shape now so that when the feed query is slow over a million rows later, you already know the fix has a name.

Part F — Webhooks: "don't call us, we'll call you"

Everything so far has the client asking and the server answering — the client always starts. But sometimes you want the reverse: you want to be told when something happens, without sitting there asking "did it happen yet? did it happen yet?" over and over. That repeated asking is called polling, and it's wasteful — most of the checks find nothing new, burning requests and battery.

A webhook flips it around. Instead of you polling someone else's server, you give them a URL of yours and say: "when the thing happens, send an HTTP request to this URL." Now they call you, once, exactly when there's news. It's the "don't call us, we'll call you" of the web — you register a callback and then stop asking.

You've received webhooks without knowing it. When a payment provider tells your shop "this order was paid," when GitHub tells a build server "someone pushed code," when a chat app posts "a new message arrived" — those are webhooks: one server making an HTTP request to another because an event occurred. It's a genuinely important pattern for connecting systems, and it's just plain HTTP pointed the other direction. (Webhooks are event notifications; when we need a truly live, two-way channel — like a feed that updates as you watch — that's WebSockets, Phase 14.)

Part G — The race that eats a vote

LinkBoard v1 works. It stores data in a file, survives restarts, serves JSON. And it contains a flaw that will corrupt your data the moment two people act at once — a flaw so important that fixing it is the reason the next big chapter of this course exists. Let's make it happen on purpose.

The setup: read, modify, write

Look at how an upvote works in storage.py. To add a point to a link, the code does three steps: (1) read the current score from the file, (2) compute new = old + 1 in memory, (3) write the whole file back. Perfectly sensible — one at a time. The trouble is that these three steps are not a single indivisible action. There's a gap between reading and writing, and in that gap, another request can sneak in.

Recall from Phase 1 that separate requests can run on separate threads at genuinely the same time. FastAPI runs our upvote handler in a thread pool, so two upvotes really do overlap. Here's the disaster, step by step, when link #1 starts at 5 points and two people upvote simultaneously:

CS fundamental: this is a race condition — the correctness of the result depends on the exact timing ("race") of concurrent operations. It's the network-and-data cousin of the shared-counter race you saw between threads in Phase 1. There it was two threads and a variable; here it's two HTTP requests and a file. Same monster, bigger stage.

The race_demo.py script fires two upvotes at the exact same instant (using a "barrier" so they truly launch together), and — because storage.py deliberately widens the read-to-write gap with a tiny sleep — the lost update happens every run, not once in a thousand. You'll cast two votes and watch the score rise by one, live in the logs.

⚠️ Why we can't just "be more careful"

The instinct is to fix this with tidier code, but you can't — the flaw is structural. As long as "read, modify, write" is three separate steps on shared data with no coordination, some interleaving will corrupt it; making the steps faster only makes the bug rarer and therefore harder to catch. The real fixes are specific mechanisms: a lock (only one upvote in the critical section at a time — Phase 1's idea), an atomic operation ("increment" as one indivisible step the storage layer guarantees), or a database transaction (Phase 6, which wraps read-modify-write so the database enforces it can't be split). A plain JSON file offers none of these, which is precisely why real applications outgrow files fast. That outgrowing is the bridge into Phase 6.

The code we wrote

Three new files in project/, and the raw-socket server.py stays as the origin story. This is the first phase where the code uses Python type hints (annotations like title: str that name what type a value should be) and decorators (the @app.get(...) lines) — the framework leans on both, so they get their own syntax colours from here on.

1. The app: routes as decorated functions (app.py)

Each route is a normal function wearing a decorator that tells the framework which method and path it answers. Compare this to Phase 3's if/elif tree — the routing is now one clean line per endpoint:

@app.get("/links")
def list_links():
    # A READ, so it's a GET. Whatever we return, FastAPI turns into JSON.
    links = storage.list_links()
    return {"count": len(links), "links": links}

@app.post("/links", status_code=201)
def submit_link(payload: LinkIn):
    # `payload: LinkIn` means: parse the JSON body into a LinkIn and VALIDATE it
    # BEFORE this runs. If title/url are missing, FastAPI 422s automatically.
    link = storage.add_link(payload.title, payload.url)
    return link   # 201 Created (Phase 3's honest "a new thing was made")

2. Validation by declaration (the Pydantic models)

We describe the shape we expect as a class, and the framework enforces it. We never write "if title is missing, return an error" — declaring the type is the check:

class LinkIn(BaseModel):
    # The body a client must send to submit a link.
    title: str = Field(min_length=1, max_length=200)
    url: str = Field(min_length=1)

Send POST /links with a missing title and you get an automatic, precise 422 error explaining exactly what was wrong — validation we'd have hand-written a dozen times in the socket server, now free.

3. Storage: the read-modify-write flaw, on purpose (storage.py)

All file access lives in one module. The upvote is the star — its three steps, and the deliberately widened gap that makes the race reproducible:

def upvote_link(link_id: int):
    data = _read()                          # STEP 1: read the whole file
    link = next((l for l in data["links"] if l["id"] == link_id), None)
    if link is None:
        return None

    old = link["points"]                    # read the score
    time.sleep(RACE_WINDOW_SECONDS)          # STEP 2: the danger window (widened)
    link["points"] = old + 1              # STEP 3: modify + write it all back
    _write(data)
    return link

4. The race demo: two votes at once (race_demo.py)

Pure standard library (so it needs no venv). It reads the score, launches two threads held at a barrier so they fire together, then reads the score again and reports whether a vote vanished:

barrier = threading.Barrier(2)          # two voters line up, then GO together
threads = [threading.Thread(target=upvote, args=(link_id, i, barrier))
           for i in range(2)]
for t in threads: t.start()
for t in threads: t.join()      # wait for both, then compare before/after

Now look at terminal 1's logs during those upvotes — the smoking gun:

14:22:01.102 [RACE] upvote id=1: read points=12 (now thinking for 0.15s — the danger window)
14:22:01.102 [RACE] upvote id=1: read points=12 (now thinking for 0.15s — the danger window)
14:22:01.255 [RACE] upvote id=1: wrote points=13 (if another upvote read 12 too, its vote is now LOST)

What this proves: both upvotes read points=12 — the identical old value — before either wrote. Both then wrote 13. The second write didn't add to the first; it replaced it. Two legitimate votes, one survived. This is the lost update of Part G, and no amount of careful coding removes it — only a lock, an atomic operation, or a database transaction does. That's the door into Phase 6.

In terminal 1, press Ctrl+C to stop uvicorn. Run deactivate to leave the virtual environment when you're done.

Break it / observe it

Optional experiments. Each pokes one idea from this phase.

  1. Watch validation reject you. Submit a link with a missing field: curl -s -X POST localhost:8000/links -H "Content-Type: application/json" -d '{"title":"no url here"}'. You get a precise 422 naming url as required — a check you never wrote. Try an empty title ("title":"") and watch min_length catch it too.
  2. Prove the race is timing, not logic. Open storage.py, set RACE_WINDOW_SECONDS = 0.0, save (uvicorn auto-reloads), and run race_demo.py a few times. Now the vote is usually not lost — the gap got too narrow to reliably interleave. The bug didn't go away; it got rarer, which is exactly what makes real races so nasty to catch. Set it back to 0.15.
  3. Feel O(n) vs O(1) in the router. Notice every request is routed instantly no matter how many routes exist — that's the framework's hash-map router (Part E). But GET /links/3 scans the list to find id 3. With three links you can't feel it; imagine a million. That felt-instant-router vs scanned-list contrast is the O(1)-vs-O(n) distinction in your own app.
  4. See serialization both ways. curl -s localhost:8000/links/1 | python3 -m json.tool shows the server serializing a Python object to JSON text. Your -d '{"title":…}' on a POST is the reverse — the server deserializing your JSON text into a validated object. You're watching Part D's round-trip from both ends.
  5. Inspect the file that is the "database." cat links.json (or open it) after a few submits and upvotes. It's just the JSON you've been reading — no magic, no database, one text file. Delete it and restart the server: it regenerates from the seed. That fragility (and the fact that a crash mid-write could corrupt it) is more motivation for Phase 6.

Glossary

New terms this phase, in plain words:

  • API (Application Programming Interface) — a stable "menu" of requests one program offers to others, hiding its internals. A web API uses HTTP requests as the menu and JSON as the responses.
  • Framework — a library that handles the repetitive plumbing (routing, parsing, validation, serialization) so you write only your app's unique logic. Ours is FastAPI; uvicorn is the server that runs it.
  • REST — a style for web APIs: name things as resources (nouns, in the path) and act on them with HTTP methods (verbs). GET /links reads; POST /links creates.
  • Resource — a "thing" the API exposes (a link, a comment), addressed by a path; sub-resources nest (/links/3/comments).
  • Serialization / deserialization — flattening a live object into bytes to send (serialize) and rebuilding it on the other side (deserialize). A serialization format is the agreed rules.
  • JSON — the web's default text serialization format: objects, arrays, strings, numbers, booleans, null. Readable and universal. XML — older, verbose, schema-strong text format. Protobuf — compact fast binary format needing a schema; for high-throughput internal services.
  • Big-O — how an operation's work grows with data size: O(1) constant, O(log n) very slow growth, O(n) linear, O(n²) quadratic (explodes).
  • Array vs linked list — contiguous storage (O(1) access by position, O(n) insert) vs pointer-chained storage (O(1) insert, O(n) access). Opposite trades.
  • Hash map — a key→value store with O(1) lookup, via a hash function that computes where a key lives instead of searching. The engine behind routers, caches, and indexes.
  • Webhook — a callback URL you register so a server calls you when an event happens, instead of you polling (asking repeatedly).
  • Race condition / lost update — a bug where the result depends on the timing of concurrent operations; a lost update is when two read-modify-writes overlap and one silently overwrites the other.
  • Type hint / decorator — a Python annotation naming a value's expected type (title: str), and a @name tag that wraps a function (@app.get(...)) — both used heavily by the framework.

Challenges & Questions

Totally optional, answers hidden — think first if you feel like it, or just reveal them.

Q1. A teammate proposes API endpoints /getLinks, /addLink, /upvoteLink?id=3, and /removeLink?id=3. Rewrite them in REST style, and explain what the REST versions gain.

RESTful versions: GET /links, POST /links, POST /links/3/upvote, and DELETE /links/3. The gains (Part C): the verb moves into the HTTP method, so the path stays a clean noun that can serve multiple actions (GET and POST on the same /links). The API becomes predictable — a client who learns /links/3 can guess GET (read), DELETE (remove) without a lookup table of verb-words. It also lets HTTP's built-in semantics do real work: GET is safe (cacheable, crawlable), DELETE is idempotent (retry-safe) — properties from Phase 3 that a /removeLink path throws away. The one id in a query string (?id=3) also becomes part of the resource path (/links/3), which reads as the actual thing being addressed.

Q2. Your app's feed loads instantly with 500 links but takes 40 seconds with 5 million. You didn't change the code. Using Part E, name the most likely culprit and the general shape of the fix.

The likely culprit is an operation whose cost grows with the data — an O(n) scan (or worse, an O(n²) nested loop) that was invisibly fast at 500 items and became unbearable at 5 million (Part E's "what happens at 1000×?"). LinkBoard's own GET /links/3 scanning the list is exactly this shape. The general fix is to replace scanning with looking up: a structure that finds items by key in O(1) or O(log n) instead of checking every one — a hash map, or in a database, an index (Phase 6's B-trees, O(log n)). If it's an O(n²) double loop, the fix is often to build a hash map of one dataset first, then do single O(1) lookups against it, collapsing O(n²) to O(n). The instinct to reach for — "stop scanning, start indexing" — is one of the most reused moves in all of performance work.

Q3. We could have made the upvote endpoint async def instead of def. The lesson says that would have hidden the race. Why — and why is a hidden race worse than a visible one?

FastAPI runs a plain def endpoint in a thread pool, so two upvotes execute on two real threads at genuinely the same time — the overlap that causes the lost update (Part G). An async def endpoint instead runs on a single event-loop thread, and cooperatively yields only at await points; since our upvote does file I/O without awaiting in the gap, two async upvotes would effectively run one-then-the-other, and the score would look correct in the demo. But the race still exists — it would reappear the moment there's an await mid-operation, or a second server process, or a real database accessed concurrently. A race you can reproduce is a race you can fix; a race that hides in testing and only strikes under production load (two servers, real concurrency) is the kind that corrupts data for weeks before anyone believes it's real. We chose the sync version precisely so the bug is honest and visible — the lesson is "concurrency bugs are timing-dependent and love to hide," which is itself the argument for structural fixes (locks, transactions) over "it passed the tests."

Q4. When would you actually reach for Protobuf over JSON, and when would reaching for it be a mistake?

Reach for Protobuf (Part D) when: two internal services exchange a very high volume of messages, where payload size and parse speed dominate, nobody needs to read the messages by eye, and you already have a build pipeline that can generate code from a schema — classic microservice-to-microservice traffic at scale. Its compactness and speed pay off, and the strict schema is a feature. It's a mistake when: the API is public or consumed by many external/unknown clients (they'd all need your schema and codegen — JSON "just works" in every language and browser), when the volume is modest (the savings are noise, but you've added a schema/build burden and lost eyeball-debuggability), or when your bottleneck is actually latency or database time, not serialization — optimizing the wrong layer. The meta-lesson: match the format to who reads it and where the real cost is, don't cargo-cult "binary is faster."

Q5. Your app needs to know when a background video finishes processing on a third-party service that can take anywhere from 10 seconds to an hour. Compare polling and a webhook for this, using Part F. Which is better and why?

Polling means asking "done yet?" on a schedule — say every 10 seconds. Over an hour that's 360 requests, almost all answered "not yet"; too-frequent polling wastes huge traffic, too-infrequent polling adds delay after it's actually done, and you must guess the right interval for a job whose length varies by 360×. A webhook fits far better here: you register a callback URL once, stop asking, and the service sends you exactly one HTTP request the moment processing finishes — no wasted polls, no guessing an interval, and minimal delay regardless of whether the job took 10 seconds or an hour (Part F). Webhooks shine precisely for infrequent, unpredictably-timed events like this. (Polling still has its place — it's simpler, needs no public endpoint on your side, and is fine for frequent, predictable checks — but "long, variable wait for a single event" is the webhook's home turf.)

Q6 (stretch). Before reading Phase 6, sketch three different ways you could stop the lost update in LinkBoard right now, and name a downside of each.

All three are real, and Phase 6 formalizes the best one. (a) A lock around the critical section — make the read-modify-write hold a lock so only one upvote runs it at a time (Phase 1's mutex idea). Correct, but it serializes upvotes (throughput bottleneck), and a single in-process lock stops helping the moment you run more than one server process (Phase 9) — you'd need a distributed lock, which is its own hard problem. (b) An atomic increment — have the storage layer offer "add 1 to this field" as one indivisible operation, so there's no read-then-write gap to interleave. Clean and fast, but a plain JSON file can't provide atomicity; you need a system that does (a database, or Redis's atomic INCR in Phase 8). (c) A database transaction — wrap the read-modify-write so the database guarantees it happens as one all-or-nothing unit, detecting or preventing conflicting concurrent updates (Phase 6's ACID). The "right" general answer, but it requires the very thing we don't have yet: a real database. Notice all three roads lead out of "a file" — which is exactly why the next chapter of LinkBoard is a database.
