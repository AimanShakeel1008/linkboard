Phase 01 — How a Computer Actually Works

Covers: M6 (Computers) and M7 (Operating Systems). You need: WSL (Ubuntu) from Phase 0, and nothing else — every demo uses plain Python 3, which Ubuntu already has. Where the demos live: concepts/phase-01-how-a-computer-works/playground/

Why this phase exists

Everything we build in this course — web servers, databases, caches, queues — runs on the same physical machine parts: a processor, some memory chips, and a storage device, all supervised by a program called the operating system.

If you understand what those parts are good at and bad at, then system design stops being a list of memorized rules. "Use a cache" stops being a rule and becomes obvious: of course you keep popular data in the fast memory instead of the slow one. "Databases write to a log first" stops being trivia and becomes obvious: of course they do — writing to the end of a file is the one thing disks are great at.

So in this phase we go all the way down to the bottom, then climb back up:

Part A — Bits and bytes: why computers only know two symbols

The light switch

Start with something from your house: a light switch. It has exactly two positions. On, or off. There is no "slightly on."

A computer chip is, at the bottom, billions of microscopic switches. Each switch is either letting electricity through, or not. Two states. That's the entire vocabulary of the machine: on or off.

We write those two states as 1 (on) and 0 (off). One such switch — one single on-or-off value — is called a bit. That's the technical name; it's short for binary digit, where "binary" just means "two options."

Why only two? Why not ten?

Fair question — we humans use ten digits (0 through 9), so why don't chips?

Because electricity in a real wire is messy. The voltage wobbles. If a chip had to distinguish ten different voltage levels to mean ten different digits, a small wobble would turn a 6 into a 7 and silently corrupt your data.

With only two levels — "clearly high" and "clearly low" — the wobble doesn't matter. High-ish still reads as 1. Low-ish still reads as 0. Two symbols is the most mistake-proof alphabet possible, so that's the one hardware uses.

From switches to numbers

One bit can only say two things: 0 or 1. Not very expressive.

But watch what happens when you line bits up. Two bits together can be in four different patterns: 00, 01, 10, 11. Three bits give eight patterns. Every bit you add doubles the number of patterns.

Line up eight bits and you get 256 possible patterns (2 multiplied by itself 8 times). A group of eight bits has its own name: a byte. The byte is the basic unit computers use to measure data — the "gram" of the digital world.

You already know the bigger units, even if you never thought about what they meant:

UnitHow bigEveryday example
1 byte8 bits, 256 possible patternsone letter of text (roughly)
1 kilobyte (KB)~1,000 bytesa paragraph of text
1 megabyte (MB)~1,000,000 bytesa small photo
1 gigabyte (GB)~1,000,000,000 bytesan hour of streamed video, roughly
1 terabyte (TB)~1,000 GBa big external drive

From numbers to everything

Here is the trick that makes computers universal: everything is just an agreement about what the numbers mean.

Text? There's a public agreement (called ASCII, later extended into Unicode) that says: the number 65 means the letter A, 66 means B, 97 means lowercase a, and so on. Text is numbers plus that agreement.

Images? Another agreement: split the picture into a grid of dots (pixels), and store three numbers per dot — how much red, green, and blue. An image is a long list of numbers plus that agreement.

Sound, video, your Python code, this very lesson file — all of it is bytes, and the only thing that makes one file a "photo" and another a "song" is which agreement (which format) the program reading it applies.

🖥️ Try it now — see the raw bytes of a real file

Open your WSL terminal and run these:

# 1. Make a tiny file containing the word "Hi"
echo -n "Hi" > /tmp/tiny.txt

# 2. Show the actual bytes inside it, as numbers
xxd /tmp/tiny.txt

You should see:

00000000: 4869                                     Hi

Reading it: 48 and 69 are the two bytes in the file, written in a compact notation called hexadecimal (more on that in one second). Byte 48 in hexadecimal is the number 72 in normal decimal — and 72 is the ASCII agreement's number for capital H. Byte 69 hex is 105 decimal — lowercase i. Two bytes, two letters. The Hi on the right is xxd being helpful and applying the ASCII agreement for you.

Ask Python to confirm the agreement:

python3 -c "print(ord('H'), ord('i'))"

Expected output: 72 105. The ord() function just looks up a character's number in the agreement. That's all text is.

About that "hexadecimal": writing long rows of bits like 01001000 is hard on human eyes, so programmers use base-16 (digits 0–9 then letters a–f) as a compact way to write bytes: one byte is always exactly two hex characters. It's purely a human convenience — the machine only ever has the bits. When you see things like 4869 or 0xFF or a git commit hash like 39f2a8b, you're looking at hexadecimal.

Part B — The CPU and the staircase of memory speeds

A very fast, very obedient, very dumb worker

The CPU (central processing unit, "the processor") is the part of the computer that actually does things. Everything else — memory, disk, network card — just stores or moves bytes. The CPU is the only part that computes.

Picture a worker at a desk who can only do extremely simple tasks: add two numbers, compare two numbers, copy a number from one place to another. Each tiny task is called an instruction.

What makes this worker useful is not intelligence — it's speed. A modern CPU executes on the order of billions of instructions per second. Your laptop's spec sheet says something like "3.2 GHz"; that's the clock speed, and it means the CPU's internal metronome ticks about 3.2 billion times per second, with (roughly) an instruction or more finished per tick.

A core is one complete worker. Modern chips pack several: a "quad-core" CPU is four workers at four desks who can genuinely do four things at the same time. Keep that number in mind — it comes back when we talk about processes and threads.

🖥️ Try it now — meet your own CPU 
lscpu | head -15

Look for two lines: Model name (your CPU's product name) and CPU(s) (how many workers you have — note this number, we'll use it later).

The problem: the worker is faster than the warehouse

Here is the single most consequential fact in computer hardware:

The CPU is much, much faster than the memory it reads from.

The worker can finish a task in a nanosecond, but fetching the materials for the task from main memory takes about a hundred nanoseconds. A worker who needs materials every task and waits 100 times the task length for each delivery would sit idle 99% of the time.

Hardware designers attacked this with a staircase of storage layers: each step down is bigger but slower. The CPU keeps the stuff it's using right now on the closest, fastest steps.

Here's the staircase. To make the speeds feel real, the last column rescales everything so that one nanosecond becomes one second — as if you were the CPU, and one step of your work took one second:

LayerWhat it isReal latency (approx.)Human scale (1 ns = 1 s)
RegistersA handful of slots inside the CPU itself~0.3 nsthe papers in your hand
L1 cacheTiny memory on the CPU core (~64 KB)~1 nsgrabbing a note off your desk — 1 second
L2 cacheBigger, slightly farther (~1 MB)~4 nsreaching a shelf behind you — 4 seconds
L3 cacheShared between cores (~tens of MB)~20 nswalking to a cabinet across the room — 20 seconds
RAM (main memory)The gigabytes your programs live in~100 nswalking to another building — 2 minutes
SSD storageYour files, persistent~100,000 ns (100 µs)a delivery truck — about 1 day
Spinning hard diskOlder persistent storage~10,000,000 ns (10 ms)a cargo ship — about 4 months
Network to another computere.g., your server calls a database~500,000–50,000,000 nsweeks to years, depending on distance

Sit with that table for a minute. On the human scale, asking RAM for data is a two-minute walk, asking the SSD is a full day's wait, and asking a spinning disk is four months.

The word cache (pronounced "cash") will follow us through the entire course, so let's define it properly here, once:

A cache is a small, fast storage layer that keeps a copy of data from a bigger, slower layer, so that repeated requests for the same data can be answered fast.

Everyday version: the notepad by your phone where you copy the five phone numbers you actually call, so you don't dig through the full address book every time. The address book is still the truth; the notepad is just a fast copy of the popular part. (Where the analogy breaks: a real cache constantly and automatically evicts old entries to make room for new ones, and a copy can go stale if the original changes — staleness is a huge topic we hit head-on in Phase 8.)

The CPU's L1/L2/L3 are hardware caches for RAM. Later, Redis (Phase 8) will be our application's cache for the database, and the design will rhyme perfectly with this staircase. Same idea, bigger scale.

One more definition while we're here, because the table used it: latency means "how long you wait for one answer." It's the length of the pipe, not its width. (Width — how much data per second — is bandwidth or throughput; Phase 2 digs into the difference.)

Part C — RAM vs storage, and the disk race

The desk and the filing cabinet

Two of the staircase layers matter most for system design, so let's give them a closer look: RAM and disk.

RAM (random-access memory, "memory") is your desk. While a program runs, its variables, its code, the web page it's building — all of it sits in RAM. It's fast (the two-minute walk, which is fast compared to the alternatives) and reasonably big (your machine probably has 8–32 GB).

RAM has one enormous catch: it's volatile, which is the technical word for forgets everything when the power goes out. Your desk gets swept completely clean every night.

Disk (storage — an SSD or hard drive) is the filing cabinet. It's huge (hundreds of GB or more) and persistent — files survive reboots and power cuts. But it is brutally slower than RAM: the day-long truck delivery, or the four-month cargo ship.

Everything that must survive — your files, your database's data — has to live on disk. Everything that must be fast has to be in RAM. A gigantic share of system design is choreographing the dance between those two facts.

🖥️ Try it now — see your desk and your cabinet 
free -h     # RAM: 'total' = desk size, 'available' = free desk space
df -h /     # disk: size and usage of WSL's Linux drive

(-h means "human-readable" — print 7.6G instead of a raw byte count.)

HDD vs SSD — and why order of access matters

There are two kinds of filing cabinet, and the difference between them reshaped system design.

A hard disk drive (HDD) is mechanical: metal platters spin at thousands of revolutions per minute, and a read arm physically swings across them, like the needle on a record player. To read a piece of data, the arm must move to the right ring of the platter and then wait for the right sector to spin underneath it. That physical repositioning is called a seek, and it takes around 10 milliseconds — an eternity (four cargo-ship months on our human scale).

A solid-state drive (SSD) has no moving parts at all — it's chips, like RAM, but persistent. No arm, no spinning, no mechanical seek. It's roughly a hundred times faster to reach a random piece of data.

Now the key insight, and it applies to both kinds of drive:

Reading data in order, front to back (sequential access) is much faster than jumping around to scattered positions (random access).

On an HDD it's obvious why: sequential means the arm barely moves and data streams under it continuously; random means paying the 10 ms seek over and over. On an SSD the mechanical excuse is gone, but sequential still wins — big orderly reads let the drive and the operating system fetch large chunks in one request instead of thousands of tiny separate requests, each with its own fixed overhead. Think of a librarian fetching one shelf in a single armload versus fetching 65,000 books one trip at a time.

This is why databases love append-only log files (Phase 6), why Kafka (Phase 11) is shockingly fast on cheap disks, and why "just add an index" (also Phase 6) is really about avoiding random hops through a giant file.

You don't have to take my word for any of this — you have a disk. Let's race it.

🏁 Demo 1: disk_race.py

The script playground/disk_race.py does exactly one honest experiment:

  1. Creates a 256 MB file of random bytes in /tmp (which lives on WSL's fast internal Linux disk — not the slower Windows /mnt/d bridge).
  2. Reads the whole file front to back in 4 KB pieces. Times it.
  3. Reads the same number of 4 KB pieces from random positions. Times that.
  4. Compares.

There's a saboteur we have to handle first, though: Linux keeps recently read file data in spare RAM, in something called the page cache — a perfect real-life example of the cache concept from Part B. The second time anything reads our test file, the data comes straight from RAM and the disk never gets involved. Great for real life, terrible for a fair disk race. So the script politely asks the kernel to forget the file from RAM before each test. (And then the --keep-cache flag lets you re-run with the cache, to watch RAM crush the disk. Two lessons, one script.)

Part D — From your code to the CPU

The translation problem

Part B said the CPU only understands tiny instructions: add, compare, copy. Those instructions are, of course, bytes — patterns of bits the chip is wired to recognize. That byte language is called machine code, and it is the only language any CPU has ever executed.

You write Python. Python is not machine code. So who translates?

There are two big strategies, and the everyday analogy is translating a recipe book written in French for a cook who only reads English:

Strategy 1 — translate the whole book once (a compiler).
Sit down, translate all of it, print an English edition. Slow to prepare, but afterwards the cook reads the English edition directly, at full speed, and the translator doesn't even need to be in the kitchen. Languages like C, C++, Go, and Rust work this way: a compiler program turns your source code into a finished file of machine code (on Windows, that's what an .exe is), and the CPU runs that file directly.

Strategy 2 — hire a live interpreter into the kitchen (an interpreter).
No preparation: the interpreter stands next to the cook, reads each French line, and speaks the English aloud as they go. Cooking starts instantly, and if the recipe changes mid-meal, no problem. But every single line pays the translation cost every single time, and the interpreter must be present in the kitchen forever. Python works this way: the program python3 is the interpreter, reading your .py file and doing what it says.

That's the honest reason "Python is slower than C": not sloppiness, but a live interpreter in the kitchen versus a pre-printed translation.

(Reality check, since you'll hear these terms: Python actually pre-chews your source into a compact internal form called bytecode first, and other languages like Java sit in clever hybrid territory. The two pure strategies above are the poles; real languages live between them.)

🖥️ Try it now — see Python's internal instructions 
python3 -c "import dis; dis.dis(lambda: 1 + 2)"

You'll see a tiny table of instructions like RETURN_CONST or LOAD_CONST — that's Python's bytecode for "produce the number 3." You just looked one level beneath your own code. (The exact instruction names vary by Python version; the point is that the table exists.)

And why should a system designer care?

Because this explains a pattern you'll see all course long: Python will never be the fastest at raw computation, and it will almost never matter. Our web app will spend its life waiting — for the database, for the disk, for the network (remember the staircase: those are the day-long and month-long waits). Shaving nanoseconds off computation while waiting months for I/O is polishing the doorknob of a house that's on fire. Where raw CPU speed does matter (databases themselves, Kafka's internals), you'll notice those systems are written in C, C++, or Java — now you know why.

Part E — Processes: programs as private apartments

From "a file of code" to "a running thing"

A program sitting on disk — python3, or your script — is just bytes. Inert. A recipe in a drawer.

When you run it, the operating system creates a process: a live, running instance of the program, with everything it needs to execute:

  • its own slice of RAM (for its variables and data),
  • an ID number — the PID (process ID) — so the OS can track it,
  • bookkeeping: which files it has open, who its parent is, how much CPU it has used.

The analogy that will carry us through two phases: a process is an apartment. It has its own private rooms (memory), its own address, its own front door. One building (your computer) holds many apartments. And crucially: no apartment can see inside another. Your browser cannot read your terminal's variables. This isolation is enforced by the OS and the hardware together, and it's why your whole computer doesn't crash when one app does — the wreckage stays inside one apartment.

Run the same program twice and you get two separate processes — two apartments that happen to have identical furniture. Same recipe, two cakes.

🖥️ Try it now — count your apartments 
ps -e | wc -l    # roughly how many processes exist right now
ps -f -u $USER   # full detail on YOURS: note the PID and PPID columns

Even an idle machine has dozens or hundreds of processes. In the second command, PPID means parent process ID — because every process is created by another process. Your shell created ps itself just to run that command (and ps died immediately after — processes can be very short-lived). It's family trees all the way up to PID 1, the first process the OS starts at boot, ancestor of everything.

🏁 Demo 2: spawn_family.py

playground/spawn_family.py makes the family tree real. The script becomes a parent and spawns three child processes (named alice, bob, and carol). The whole family then stays alive for about a minute, printing heartbeats — deliberately long enough for you to open a second terminal and inspect them from outside with ps, pstree, and top, like a landlord checking the building register.

It also proves the apartments are really private: parent and children each write a different value into a variable with the same name, and nobody's write ever affects anybody else. Hold that thought — Part F shows the exact opposite.

Context switching: one chef, many dishes

Here's a puzzle from the numbers you've collected: lscpu said you have maybe 8 cores, but ps -e counted hundreds of processes. Hundreds of apartments, eight workers. How does everything seem to run at once?

It doesn't. The OS plays a shell game. Each core runs one process for a short slice of time (a few milliseconds), then the OS pauses it, saves its complete working state — every register, its exact position in the code, like photographing a chessboard mid-game — loads the saved state of a different process, and runs that one for a slice. This swap is called a context switch, and it happens thousands of times per second. The illusion of "everything at once" is just very fast turn-taking.

Everyday version: one chef cooking six dishes by rotating between pans. Every rotation costs a little time — put down the knife, wash hands, reread the recipe, remember where you were. Context switches likewise aren't free: a few microseconds each, plus a sneakier cost — the CPU caches from Part B are full of the old process's data, useless to the new one, so the new process starts its turn with a cold desk.

File this away: "too many things taking turns" is a real failure mode we'll meet again (it's part of why "just add more threads" stops working, Phase 9 territory).

Part F — Threads: roommates who share everything

Apartments vs roommates

Sometimes you want two flows of work that cooperate closely — share the same data, work on the same job. Two separate processes are awkward for this: private apartments mean any sharing requires formally mailing data between them.

Enter the thread. A thread is a flow of execution inside a process. Every process has at least one thread (the "main" thread). It can start more. The defining property, and the entire reason threads exist:

All threads in a process share that process's memory.

Threads are roommates in one apartment. Same rooms, same fridge, same whiteboard on the wall. No mailing anything — roommate A writes on the whiteboard and roommate B just... sees it. Instantly. That's wonderfully convenient and fast.

And it is the most dangerous convenience in programming.

The whiteboard disaster

Two roommates keep a shared tally on the whiteboard — say, money collected for rent. Currently it reads 41. Both collect $1 at the same moment.

  • Roommate A looks at the board. Reads 41.
  • Roommate B looks at the board, a split second later. Also reads 41. (A hasn't written yet — A is still doing the math.)
  • A computes 41 + 1 = 42. Writes 42.
  • B computes 41 + 1 = 42. Writes 42.

Two dollars were collected. The board went up by one. A dollar just vanished — no crash, no error, no smoke. The board simply lies now.

This bug has a name: a race condition — the final result depends on the accidental timing of who read and wrote when. The nasty parts:

  1. It's invisible in the code. counter = counter + 1 looks like one safe action. It's secretly three: read the value, add one, write it back. Any other thread can sneak in between those steps.
  2. It's intermittent. Most of the time the threads happen not to collide, everything works, and your tests pass. Then production traffic arrives, collisions start, and data quietly corrupts.

This is the most important bug class in this course. In Phase 4, two simultaneous upvotes on a LinkBoard post will eat each other in exactly this way. In Phase 6, database transactions will slay it properly. Today you just get to watch it happen.

🏁 Demo 3: thread_garble.py

playground/thread_garble.py stages the whiteboard disaster with real threads: two threads each add 1 to a shared counter 5,000 times. The correct final answer is 10,000.

Round 1 runs with no protection. You'll get some number far below 10,000 — thousands of increments silently destroyed. (One honesty note: the script inserts a deliberate micro-pause between each thread's read and write. Real code has no such pause, but the read-write gap always exists — it's just nanoseconds wide, so collisions are rare. Rare is worse: rare means "passes every test, fails on Black Friday." We widen the gap so you can watch the crime in seconds rather than wait weeks for it.)

Round 2 runs the same job with a lock (also called a mutex, for mutual exclusion) — the bathroom-door rule: one roommate inside at a time, everyone else waits at the door. Result: exactly 10,000, every time. Also: slower. Taking turns costs speed. Correct-but-slower versus fast-but-wrong is a tradeoff you will negotiate for the rest of this course, all the way up to distributed databases (Phase 15).

Part G — The kernel: the building superintendent

Who's been doing all this, exactly?

Throughout this lesson someone has been quietly doing things: creating processes, pausing and resuming them, shuttling data between RAM and disk, keeping apartments private. That someone is the operating system — and its core, the part that holds all the real power, is called the kernel.

The analogy that fits best: the kernel is the building superintendent. Tenants (processes) don't rewire electricity or cut new keys themselves — imagine the chaos if any tenant could. They request things from the superintendent, who holds the master keys and decides what's allowed.

Concretely, the kernel is the only code on the machine with direct authority over hardware. It decides which process runs on which core (scheduling, the context-switch shell game), who gets memory, who may open which file, and it operates every device — disk, network card, keyboard — on the tenants' behalf.

This power split is enforced by the CPU itself, which runs in one of two modes: kernel mode (all powers — only the kernel runs here) and user mode (restricted — everything else, including every program you've ever written). In user mode, the dangerous instructions simply don't work. It's not a convention; it's wiring.

System calls — the doorbell

So if your Python script can't touch the disk... how did disk_race.py read a file?

It rang the doorbell. A system call ("syscall") is the formal mechanism by which a user-mode process asks the kernel to do something on its behalf. Your program prepares its request ("read 4096 bytes from this file"), triggers a special CPU instruction, the CPU flips into kernel mode at a fixed, kernel-controlled entry point (you can't jump into the middle of the super's office — you ring the bell and the super answers), the kernel checks permissions, does the work, and hands back the result as it returns you to user mode.

Every interaction with the world outside your process is a syscall. Reading files, sending network data, printing to the terminal, starting a process, even asking the time. A typical program is a thin layer of your logic wrapped around a torrent of syscalls — and you can watch the torrent:

🖥️ Try it now — spy on a program's syscalls 
sudo apt install -y strace     # one-time install of the spy tool
strace -c ls /tmp

The first command installs strace; the second runs plain old ls with the spy attached. After ls's normal output you'll get a table: every syscall it made, how many times, how long they took. A boring one-line ls makes dozens of syscalls — openat (open a file/directory), read, write (that's it printing!), close, mmap (memory requests)...

For the full firehose, try strace ls /tmp (no -c) — every individual syscall, live. That's the true conversation between a program and the kernel. When a tool in a later phase feels mysterious, strace is how we'll de-mystify it.

Virtual memory — the kindest lie in computing

One more superintendent trick, and it's a beautiful one.

Every process believes it has a huge, continuous range of memory addresses all to itself, starting from zero. Alice's process uses "address 5000." So does Bob's. Same number — no conflict. How?

Because both are being lied to. Those addresses are virtual — apartment room numbers, not warehouse coordinates. Every apartment has a "room 1," and no tenant ever knows or needs to know where their rooms physically sit. The kernel (with a dedicated helper circuit in the CPU) keeps a translation table per process — this process's room 5000 = that physical RAM location — and translates every single memory access, transparently.

This one lie buys three huge wins:

  1. Isolation for free. Your addresses simply cannot name another process's memory — the translation table has no entry for it. The apartment privacy from Part E isn't politeness; it's this mechanism. And when a process uses a virtual address that maps to nothing? The kernel kills it with the infamous segmentation fault — the super catching a tenant trying a door that doesn't exist.
  2. Simplicity. Every program can be written as if it owns a clean, empty memory space. No program ever coordinates with any other.
  3. Flexibility. Memory is managed in fixed-size chunks called pages (typically 4 KB — the same 4 KB our disk demo used; it's the kernel's favorite unit). Pages a process hasn't touched lately can even be evicted to disk to free up RAM (swapping). Sounds clever — but remember the staircase: disk is the day-long truck. A machine forced to swap heavily slows to a crawl, thrashing. "The server is swapping" is ops-speak for we're about to have a very bad day; watching for it comes in Phase 16.

And a two-sentence preview of file systems, the kernel's other great illusion: a disk is really just billions of numbered 4 KB-ish slots, and "files with names in folders" is a data structure the kernel maintains on top of those slots — a card catalog over a warehouse. That card catalog metaphor returns with force when we hit database indexes in Phase 6.

Part H — Blocking I/O: the idea that explains web servers

First, the letters: what is I/O?

I/O stands for input/output — any time a program talks to the world outside its own memory. Reading a file is I/O. A network request is I/O. Printing to the terminal is I/O. (Each one, you now know, is a syscall.)

Now put two facts you've already learned side by side:

  • CPU work happens in nanoseconds.
  • I/O takes microseconds (SSD), milliseconds (network)... on the human scale: days, months.

So the defining feature of I/O is: the answer takes ages to arrive compared to computation. Which raises the question — what does your program do while the disk truck is driving?

It stops. That's blocking.

When your Python calls f.read() — a syscall — the kernel sees the data isn't ready yet, and it doesn't just leave your process spinning. It marks your process as waiting, takes it off the CPU entirely, and runs someone else (there are hundreds of other processes; the chef rotates to another pan). When the disk finally delivers, the kernel marks your process runnable again, and it resumes at the exact line it stopped.

From the outside: your program hit f.read() and froze until the data arrived. That freeze is called blocking. The call blocks.

Efficient for the machine as a whole — no CPU is wasted on waiting. But notice what it means for your program: while blocked, your program is doing nothing. Not "working slowly." Nothing.

The frozen waiter

Now the analogy this whole part is named for. Picture a restaurant with one waiter, run under a strange rule: after taking a table's order and handing it to the kitchen, the waiter must stand frozen at that table until the food comes out, then serve it — and only then may he approach the next table.

Kitchen takes ten minutes per dish. Five tables. The fifth table waits nearly an hour just to order, while the waiter spends the whole hour standing perfectly still. The waiter isn't slow. The kitchen isn't even slow, really. The rule is the disaster: the waiting doesn't overlap.

That waiter is a naive web server. "Take request, wait for the database, respond, only then look at the next customer" — one slow request freezes every user behind it. In Phase 2 we will build exactly this server, on purpose, from raw sockets, and you will be the ignored customer. It's infuriating. It's also the founding problem of server architecture:

  • Hire more waiters — a thread or process per customer (Phase 2's fix, and its limits in Phase 9).
  • One brilliant unfrozen waiter who drops off orders and keeps moving, checking back when kitchen bells ring — non-blocking, event-driven I/O, the trick behind Node.js and Python's asyncio (properly explored when we hit real-time systems, Phase 14).

Where the analogy breaks, one more time: a blocked process costs the machine almost nothing (the kernel runs other work meanwhile) — a frozen human waiter is pure waste. The waste in software is the customers' time, not the CPU's.

🏁 Demo 4: blocking_waiter.py

playground/blocking_waiter.py makes you feel the arithmetic. It "fetches three web pages," each fetch being two seconds of pure blocked waiting.

  • Round 1 — one waiter: fetches them one after another. The waits stack up: ~6 seconds.
  • Round 2 — three waiters: one thread per fetch; all three block at the same time, all three finish together: ~2 seconds.

Same work, same CPU, three times faster — because the time was never being spent on the CPU. It was spent waiting, and waiting overlaps for free. Every timestamped log line shows you exactly which waits stacked and which overlapped. This is why threads help I/O-heavy programs even on a single-core machine — and this experiment is the seed of everything in Phases 2, 9, and 14.

Demo 3 — the race condition

python3 thread_garble.py

Expected output shape:

[DEMO] === Shared-memory race condition demo ===
[ROUND] --- ROUND 1: no lock (the race) ---
[ROUND] two threads will each add 1 to a shared counter 5,000 times
[ROUND] if nothing goes wrong, the counter should end at 10,000
[THREAD-A] finished my 5,000 increments (unsafe)
[THREAD-B] finished my 5,000 increments (unsafe)
[ROUND] both threads done in 0.31 s — final counter: 5,127
[RESULT] expected 10,000 but got 5,127 — 4,873 increments VANISHED (49% lost!)
[RESULT] no crash, no error message — the updates are just gone. This is why race conditions are so feared.

[ROUND] --- ROUND 2: with a lock (the fix) ---
   ...
[ROUND] both threads done in 0.52 s — final counter: 10,000
[RESULT] exactly 10,000, every single increment survived — the lock forced the threads to take turns
[RESULT] the price: taking turns is slower. Correctness costs something. ...

What the lines prove:

  • Round 1's final counter (any number well below 10,000 — often close to half) — thousands of read-modify-write collisions, each destroying an update. No error was raised at any point. The program "worked."
  • Round 2: exactly 10,000, every run, forever — mutual exclusion turns three fragile steps into one uninterruptible unit. Note the higher time: the corrupted version was faster. Wrong answers usually are.

Run Round 1 a few times — the lost count changes every run. That non-repeatability is the signature of a race, and it's why they're hell to debug from bug reports: "it only fails sometimes" is this.

Break it / observe it

Optional experiments — each one deepens a concept by poking it. Nothing here can damage anything.

  1. Break the ASCII agreement. Interpret a binary file as if it were text: head -c 300 /usr/bin/ls | xxd shows machine code bytes (this is Part D's compiled machine code, in the flesh — ls is a compiled program!). Now let the terminal misinterpret a text-adjacent slice: head -c 64 /usr/bin/ls prints garbage symbols. Same bytes, wrong agreement = nonsense. That's all "file corruption" or mojibake ever is.
  2. Cache-warm a grep. Pick a big-ish file and time the same search twice: time grep -r "nonexistentword" /usr/share/doc > /dev/null — run it two times in a row. Second run is typically much faster: page cache again, no script needed. (time prefixed to any command reports how long it took — your new favorite tool.)
  3. Watch a context-switch counter. grep ctxt /proc/$$/status shows how many times your shell has been switched off the CPU ($$ means "my shell's PID"; /proc is a fake folder where the kernel publishes its bookkeeping as readable files — the super's clipboard, public edition). Voluntary switches = it blocked (waiting for your keystrokes!); involuntary = the OS yanked it mid-slice. Run it, press enter a few times, run it again — watch voluntary climb.
  4. Make the race disappear. In thread_garble.py, delete the time.sleep(0) line in unsafe_worker and re-run. On modern Python, Round 1 may now come out correct (or nearly) every time — the collision window shrank from "always" to "almost never." The bug is still there. Sit with how terrifying that is: the identical bug, and every test now passes. (Put the line back after.)
  5. Strace the disk race. strace -c python3 disk_race.py — in the final table, find pread64 with ~65k+ calls: every one of those was a doorbell ring to the kernel (Part G) for 4 KB of file (Part C). Two lesson parts, one table.
  6. Kill a child, orphan nothing. Start spawn_family.py, and from terminal 2, kill <alice's pid>. Alice dies instantly (her heartbeats stop); bob and carol keep beating, and the parent reports alice's exit code as negative (killed by signal). Apartments: one demolished, neighbors unharmed. Compare Phase 0's kill on the runaway — same tool, now you know what it's really doing.

Glossary

New terms this phase, in plain words:

  • Bit — one on/off value; the smallest unit of data. Byte — 8 bits.
  • Binary / hexadecimal — writing numbers with 2 symbols (what hardware does) / with 16 (a compact human notation for bytes).
  • ASCII / Unicode — the public agreements mapping numbers to characters.
  • CPU / core / clock speed — the part that computes / one independent worker in it / how many billion ticks per second it runs at.
  • Instruction / machine code — one primitive CPU operation / the raw byte language of instructions, the only language a CPU runs.
  • Register — one of a handful of storage slots inside the CPU itself.
  • Cache — a small fast layer keeping copies of popular data from a big slow layer. L1/L2/L3 — the CPU's built-in caches. Page cache — the kernel's in-RAM cache of file data.
  • Latency — how long one answer takes to come back.
  • RAM — fast, volatile working memory. Volatile — erased at power loss. Persistent — survives power loss.
  • HDD / SSD — mechanical spinning storage / chip-based storage. Seek — an HDD's physical repositioning to reach data.
  • Sequential vs random access — reading in order vs jumping around; in order is drastically faster on disks.
  • Compiler / interpreter / bytecode — translate the whole program ahead of time / translate line by line while running / a compact internal form interpreters pre-chew source into.
  • Process / PID / PPID — a running instance of a program with private memory / its ID number / its parent's ID.
  • Context switch / scheduling — the OS swapping which process a core runs / the OS's policy for those turns.
  • Thread — a flow of execution inside a process; shares the process's memory with its sibling threads.
  • Race condition — a bug where the result depends on accidental timing of concurrent reads/writes; intermittent and silent.
  • Lock (mutex) — a take-turns mechanism: one thread at a time in a critical section.
  • Kernel / user mode / kernel mode — the OS core with hardware authority / the restricted CPU mode everything else runs in / the all-powers mode only the kernel runs in.
  • System call (syscall) — the doorbell: a user program's formal request for a kernel service.
  • Virtual memory / page / swapping / segmentation fault — per-process fake addresses translated to real RAM / the 4 KB unit memory is managed in / evicting idle pages to disk / the kill a process gets for touching an unmapped address.
  • I/O — any input/output: files, network, terminal.
  • Blocking — a call that freezes your program until its result arrives, while the kernel runs others.

Challenges & Questions

Totally optional, answers hidden — think first if you feel like it, or just reveal them.

Q1. Your phone has 8 GB of RAM and 256 GB of storage. Using this phase's analogies: which is the desk, which is the filing cabinet, and why does closing apps "free up memory" but not "free up space"?

RAM is the desk, storage is the filing cabinet. A running app's working data lives in RAM (on the desk); closing the app ends its process, and the kernel reclaims all its RAM — clearing the desk. But the app's files (the app itself, your photos, its saved data) live in storage and aren't touched by closing it — the cabinet stays full. "Free up space" means deleting files from the cabinet; "free up memory" means clearing desk space by ending processes. Two different resources, two different verbs.

Q2. The staircase table says a network hop can cost milliseconds — months on the human scale. LinkBoard will eventually call a database over the network on every request. Using ONLY ideas from this lesson, guess two strategies to keep that from making every page slow. (You're allowed to invent — both real answers are hiding in this lesson.)

1. Cache the popular answers closer — keep copies of frequent query results in the app's own RAM or a nearby fast store, so repeat requests skip the network trip entirely. That's the cache concept (Part B), scaled up: it becomes Redis in Phase 8. 2. Overlap the waiting — a network call is blocking I/O; while one request waits on the database, the server should be working on other requests, exactly like Round 2 of blocking_waiter.py. That's threads/async, arriving in Phase 2. (Bonus third answer: send fewer/bigger requests instead of many tiny ones — the sequential-vs-random lesson in network form; that becomes the N+1 problem in Phase 6.)

Q3. In thread_garble.py, the lock made Round 2 correct. Why not make the lesson simpler by having threads that never share anything? What would we lose? (Hint: Part E built exactly that.)

Threads that share nothing are... basically processes — that's the apartment model, and we have it (Part E, spawn_family.py). What sharing buys: instant, copy-free access to common data (one whiteboard instead of mailing letters between apartments) and cheap coordination on one job. Web servers, for example, want many workers sharing one pool of database connections and one in-memory cache. The whole tradeoff: processes = safe-but-awkward sharing, threads = effortless-but-dangerous sharing. Systems pick per situation, and lots of real servers use both at once (Phase 9's worker models).

Q4. A friend says: "Python is interpreted, so a Python web app will feel slow to users; rewrite it in C++." Using the staircase and Part D, give the honest counterargument — and name one situation where the friend is right.

A web app's request time is dominated by I/O waits: network hops to the database (milliseconds — months on the human scale) dwarf the microseconds of extra interpreter overhead. Making computation 50x faster shaves a rounding error off a database-bound request; users can't feel it. The friend is right where work is genuinely CPU-bound: heavy number crunching, encoding video, the internals of a database or message broker itself — which is exactly why Postgres and Kafka aren't written in Python, while the apps calling them happily are.

Q5. Experiment 4 removed the sleep(0) and the race "went away." Your team's tests all pass. Argue, in two sentences, why you'd still demand the lock before shipping — then say which future phase inherits this exact argument.

The read-modify-write gap still exists — it's merely nanoseconds wide, so collisions moved from "constant" to "rare," and rare collisions on a busy production server mean corrupting data occasionally and untraceably, which is strictly worse than failing loudly in tests. Correctness can't rest on lucky timing. Phase 6 inherits this argument verbatim: database transactions are the disciplined, industrial-strength version of "take the lock before the read-modify-write" (and Phase 4 will first show the bug live, in real HTTP traffic).

Q6 (stretch). strace -c ls showed dozens of syscalls. Each syscall has overhead (mode switch, permission checks). Predict: for a program reading a 1 GB file, which is faster — 262,144 reads of 4 KB, or 1,024 reads of 1 MB? Connect your answer to BOTH the disk race and the sequential-access lesson.

The 1,024 big reads win. Same total bytes, but 256x fewer doorbell rings — each syscall's fixed overhead (mode switches, checks) is paid 1,024 times instead of 262,144. This is the sequential-vs-random insight in a second costume: fixed per-request overhead makes many-small-requests lose to few-big-requests, whether the overhead is a disk seek, a syscall, or (coming soon) a network round-trip. Recognizing that same shape in new costumes is half of system design — you'll spot it in HTTP connection reuse (Phase 3), the N+1 query problem (Phase 6), and message batching in Kafka (Phase 11).
