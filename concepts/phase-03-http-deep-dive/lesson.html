Why this phase exists

In Phase 2 we built a web server out of raw sockets and got a page back. But look closely at what that server actually understood about the web: almost nothing. It read the very first line, snatched the method and path, threw away every header, spoke only GET, and hung up after a single reply. It was a machine that could receive an HTTP request the way a dog can receive a newspaper — it fetched the object without reading a word of it.

This phase is where we read the newspaper. The thing riding on top of TCP — the actual language your browser and a server speak — is called HTTP (HyperText Transfer Protocol). It is the single most important protocol you will touch in this entire course, because everything LinkBoard does from here on is an HTTP conversation: submitting a link, logging in, loading the homepage, calling an API, talking to another service. If HTTP is a black box to you, all of web development stays a black box. So we open it all the way up.

And here's the reassuring truth, the same one as every phase: HTTP is not magic and it is not even complicated. HTTP is just structured text sent over a socket. A request is text in a strict shape. A response is text in a strict shape. That's the whole protocol. By the end of this phase you'll have written the parser that reads that text and the builder that writes it — by hand — and you'll never again wonder what a "404" or a "header" or a "cookie" really is.

Part A — Request & response, and the word "stateless"

HTTP has exactly one shape of interaction, and it never varies: the client asks, the server answers, done. One request in, one response out. The client (your browser, curl, LinkBoard's future JavaScript) always speaks first; the server only ever replies. A server never spontaneously sends you something out of the blue over plain HTTP — it waits to be asked. (Making the server able to push updates on its own is a real effort that needs a different tool, WebSockets, which is Phase 14.)

"Stateless": the server has amnesia between requests

Here is the single most important — and most surprising — fact about HTTP: it is stateless. That word gets thrown around a lot, so let's make it concrete. Stateless means the server, by default, remembers absolutely nothing about you from one request to the next. Each request arrives as if it were the very first time the server has ever heard from you. There is no built-in "who is this and what were they doing?"

Picture a coffee shop where the barista has total amnesia the instant you step away from the counter. You order a coffee, walk to find a seat, come back for a napkin — and the barista has no idea you were ever there. "Can I help you?" Every single interaction starts from zero. That is an HTTP server. Your first request ("log me in") and your next request ("show my profile") are, to the server, two strangers who happen to look alike.

📝 Why on earth would you design it that way?

It sounds like a flaw, but statelessness is a superpower for scaling — and it's why the whole later half of this course is even possible. If the server keeps no memory of you, then any server can handle any request. There's no "but only server #3 remembers this user." That means you can put a hundred identical servers behind a load balancer (Phase 9) and each request lands on whichever one is free. The cost of that freedom is that we must add the memory back on top, deliberately, when we want it — which is exactly what cookies and sessions (Part F) are for. Amnesia by default; memory by choice.

Quick check: if HTTP forgets you after every request, how does a site keep you logged in across dozens of page loads?

The server stays amnesiac, so the memory has to travel with each request. On login the server hands your browser a small token (a cookie), and your browser automatically attaches that token to every future request. So each request arrives carrying its own proof of "I'm the user who logged in a minute ago." The server still remembers nothing on its own — it just reads the note you bring every time. That's Part F, and it's the workaround the entire logged-in web is built on.

Part B — Anatomy of a request: it's just structured text

Let's dissect a real request. When you run curl -v localhost:8000/about, this is (roughly) the exact text your machine sends across the socket. The invisible line-endings are shown as \r\n (a carriage-return followed by a line-feed — a two-character line break HTTP inherited from typewriters and still insists on):

GET /about HTTP/1.1          # 1. the REQUEST LINE: method, path, version
Host: localhost:8000         # 2. HEADERS: one "Name: value" per line
User-Agent: curl/7.81.0
Accept: */*
                             # 3. a BLANK LINE — "the headers are over"
                             # 4. the BODY goes here (GET usually has none)

Four parts, always in this order. Learn these four and you have learned the request half of HTTP:

  1. The request line — three words: the method (the verb: what do you want to do?), the path (which resource? /about), and the version (HTTP/1.1). This one line is the essence of the request.
  2. The headers — a list of Name: value lines carrying metadata about the request: who's asking (User-Agent), what host they meant (Host), what formats they'll accept (Accept), how long the body is (Content-Length), and much more. Header names are case-insensitive (Host and host are the same). We'll spend Part E here.
  3. The blank line — a single empty line. This is not decoration; it is a hard rule. It's how the server knows the headers have ended and the body (if any) begins. Our Phase 2 server literally looped reading bytes until it saw this blank line (\r\n\r\n).
  4. The body — the actual payload, present only on requests that carry data (a form submission, a file upload, a JSON API call). A plain GET has no body; a POST that submits a link does. The Content-Length header tells the server exactly how many body bytes to read.

The response has the mirror-image shape, which you already saw hand-built in Phase 2: a status line instead of a request line (HTTP/1.1 200 OK), then headers, then a blank line, then the body (the HTML page). Same four parts, same order. Request and response are two envelopes of the identical design.

"Explain what happens in an HTTP request" is one of the most common opening questions in a web or systems interview, precisely because it's a fractal — you can answer it in ten seconds ("method, path, headers, body; server replies with a status line, headers, body") or spend an hour going down through TCP, DNS, and TLS. Knowing the four parts cold, and being able to name a header for each concern (auth, caching, content type), is the difference between "I use a framework" and "I understand the wire." You're building the second kind of knowledge by writing the parser yourself.

Part C — The methods, and the idea of idempotency

The method is the verb of the request — it announces your intent. The path says which resource; the method says what to do with it. There are a handful, and each has an agreed meaning that the whole web honors:

MethodMeansHas a body?Example
GETread a resource; never change anythingnoload the homepage
HEADlike GET but headers only, no bodyno"is this page there / how big?"
POSTcreate something / submit datayessubmit a new link
PUTreplace a resource with this exact valueyesset a link's points to 100
PATCHpartially update a resourceyeschange just the title
DELETEremove a resourceusually nodelete a link

The two rules that give methods meaning: "safe" and "idempotent"

Two properties separate these verbs, and both matter enormously in real systems. They sound like jargon; they're actually simple promises.

A method is safe if it only reads — it never changes anything on the server. GET and HEAD are safe. This is why a browser feels free to pre-fetch links, and why a search-engine crawler can hit every GET on your site without fear: reading can't break anything. It's also why you must never hide a destructive action behind a GET (a link that deletes a post when clicked is a classic, painful bug — a crawler or a "prefetch" will find it and quietly delete everything).

A method is idempotent if doing it once and doing it many times in a row leave the server in the same state. The word looks scary; the idea is everyday. Think of a light switch labeled "OFF." Flip it to OFF once, or flip it to OFF ten times — the light is off either way. That's idempotent: repeating the action doesn't pile up. Now think of a switch labeled "PRESS TO ADD ONE COOKIE TO MY ORDER." Press it once, you have one cookie; press it five times, you have five. That's not idempotent: each repeat changes the world again.

Here's the idempotency of each method, and — crucially — why you care:

  • GET, HEAD, PUT, DELETE are idempotent. PUT sets a resource to a specific value (do it twice, same value). DELETE removes it (delete twice, it's still gone — same end state). Reads obviously don't change anything.
  • POST is not idempotent. Each POST /submit adds a link. Send it twice and you get two links.
📝 Why this is a real engineering concern, not trivia

Networks are unreliable (Phase 2 — packets get lost). So imagine a client sends a request, and the reply never comes back. Did the server do the work and only the reply got lost? Or did the whole request get lost? The client can't tell. Its only options are "retry" or "give up." If the method is idempotent, retrying is completely safe — worst case the server does the same harmless thing again. If it's not (a POST that charges a credit card), a blind retry might charge you twice. This is exactly why the "are you sure you want to resubmit this form?" browser warning exists, and why serious systems put special machinery around non-idempotent operations (idempotency keys — a Phase 21 topic). The instant you internalize "idempotent means retry-safe," a huge amount of distributed-systems design suddenly makes sense.

Quick check: your app has a "like" button. Should clicking it be a GET, and is "toggle like" idempotent?

It should not be a GET — liking changes server state, and GET must be safe (read-only), or a crawler/prefetch could "like" things by merely loading links. Use POST (or PUT). Is "toggle" idempotent? No — toggling once turns the like on, toggling again turns it off; repeats flip back and forth, so the end state depends on the count. If instead you design it as "set liked = true" (a PUT-style set), it becomes idempotent: any number of repeats leaves it liked. Choosing "set to a value" over "toggle" to gain idempotency is a real API-design move you'll make on purpose.

Part D — Status codes: the server's three-digit verdict

Every response starts with a status code — a three-digit number that tells the client how the request went, instantly, before it reads a single byte of the body. You met a few in Phase 2 (200, 404, 405). Now let's see the whole system, because it's beautifully organized: the first digit is the family (the gist), and the rest is the specific case.

The families, in plain words:

  • 1xx — Informational. "Message received, keep going." Rare in everyday work; you can mostly ignore them.
  • 2xx — Success. It worked. 200 OK (here's what you asked for), 201 Created (your POST made a new thing), 204 No Content (worked, but there's deliberately nothing to send back).
  • 3xx — Redirection. "What you want is elsewhere." 301 Moved Permanently (update your bookmarks) and 302 Found (temporary) both come with a Location header naming the new address; the browser follows it. 304 Not Modified is the caching star of Phase 8: "your saved copy is still good, don't re-download."
  • 4xx — Client error. You messed up. 400 Bad Request (we couldn't parse what you sent), 401 Unauthorized (you're not logged in), 403 Forbidden (we know who you are; you still can't), 404 Not Found (no such thing), 405 Method Not Allowed (wrong verb for this path), 429 Too Many Requests (slow down — rate limiting).
  • 5xx — Server error. The server messed up. 500 Internal Server Error (a bug threw an exception), 502 Bad Gateway (a server in front couldn't reach the one behind it), 503 Service Unavailable (overloaded or down for maintenance).
⚠️ The 401 vs 403 trap (and the 401 misnomer)

These two are constantly confused. 401 Unauthorized actually means "unauthenticated" — the server doesn't know who you are; log in and try again. 403 Forbidden means "I know exactly who you are, and you're still not allowed" — logging in won't help; you lack permission. (Yes, "401 Unauthorized" is a famous naming mistake baked into the standard forever — it's about authentication, not authorization.) A related judgment call: some sites return 404 instead of 403 on purpose, so an attacker can't even learn that a protected resource exists. These distinctions are Phase 7 (auth) and Phase 24 (security) material, but you're meeting the vocabulary now.

Quick check: your server catches a bug in your own code while handling a request. Is that a 4xx or a 5xx?

A 5xx — specifically 500 Internal Server Error. The 4xx family is for when the client sent something wrong; a bug in your handler is the server's fault, so it belongs in the 5xx family. This matters operationally: dashboards alarm on 5xx rates (your fault, wake someone up) very differently from 4xx rates (often just clients probing bad URLs). Our upgraded server does exactly this — it wraps each route in a try/except and turns any handler crash into a 500 instead of dying, which you'll see in the code walkthrough.

Part E — The headers that actually matter

Headers are the metadata of HTTP — the Name: value lines that ride alongside every request and response. There are dozens, but a working engineer really only needs to know a couple dozen, and a beginner needs maybe ten. Here are the ones you'll meet again and again, grouped by what they're for:

HeaderDirectionWhat it does
Hostrequestwhich website you meant (one server/IP can host many sites; this disambiguates). The one header HTTP/1.1 requires.
User-Agentrequestwhat software is asking (browser, curl, a bot). Servers use it for stats and quirks.
Acceptrequestwhat formats the client can handle (text/html, application/json…). The server may honor it.
Content-Typebothwhat format the body actually is (text/html, application/json, application/x-www-form-urlencoded).
Content-Lengthbothhow many bytes the body is, so the other side knows when it's fully arrived (Phase 2's crucial number).
Authorizationrequestyour credentials / token — "here's proof of who I am" (Part F, and Phase 7).
Locationresponseon a 3xx redirect, the address to go to instead.
Cache-Controlbothif/how long this may be cached (no-store, max-age=60…). The whole language of Phase 8.
Set-Cookieresponse"store this and send it back to me next time" — how state is added (Part F).
Cookierequestthe stored cookies the browser echoes back on every request.
Connectionbothkeep-alive or close — reuse this TCP connection, or hang up after? (Part G).

Notice the pattern: headers are how the two sides negotiate everything that isn't the raw content itself — identity, format, length, caching, connection reuse, authentication. The request line says what you want; the headers say everything about how. Our server reads Host, User-Agent, Content-Length, Authorization, Cookie, and Connection for real, and writes Content-Type, Content-Length, Location, Cache-Control, Set-Cookie, and Connection — you'll watch each one do its job.

Part F — Cookies: how a stateless protocol remembers you

Back to Part A's puzzle: HTTP is stateless — the server forgets you between requests — yet you obviously stay logged in as you click around. The trick is beautifully simple, and it's called a cookie.

A cookie is a small piece of text the server asks the browser to hold onto and hand back. The flow is a three-step loop:

  1. On some response, the server includes a Set-Cookie: visits=1 header. This says: "please remember this name=value."
  2. The browser stores it, quietly, tied to that site.
  3. On every subsequent request to that site, the browser automatically adds a Cookie: visits=1 header. The server reads it and — presto — "remembers" you, even though it kept no memory of its own.

Our upgraded server has a live demo of exactly this at /count: it reads the visits cookie, adds one, and sends back a new Set-Cookie. Hit it repeatedly with the same cookie jar and watch the count climb — the server "remembering" you without remembering anything.

What actually goes in the cookie? Sessions vs JWT

A login cookie shouldn't literally say user=admin — anyone could just edit it. There are two mainstream designs for doing it safely, and you'll choose between them in Phase 7. Meet them by name now:

  • Session cookie (server-side memory). The cookie holds only a big random meaningless ID, like session=8f3a…c2. The real data ("this ID is user 42, logged in at 3pm") lives in a table on the server (or in Redis — Phase 8). Each request, the server looks up the ID to find who you are. The cookie is just a claim-check ticket; the coat is in the server's cloakroom. Easy to revoke (delete the row), but every request needs a lookup.
  • JWT (self-contained token). The cookie (or an Authorization header) holds the actual data — "user 42, expires 4pm" — plus a cryptographic signature the server can verify but a client can't forge. No server-side lookup needed: the token carries its own proof. Great for scaling (any server can verify it alone), but awkward to revoke before it expires. This is JSON Web Token, and it leans on the signing ideas of Phase 7.

The tradeoff — a lookup-required ticket you can revoke instantly, versus a self-proving token that scales but lingers — is a genuine design decision, not a right-vs-wrong. We'll implement one for real when LinkBoard gets logins. For now the point is: a cookie is just text the browser carries for you, and what you put in it is where the engineering lives.

⚠️ Cookies are why security gets interesting

Because the browser sends cookies automatically on every request to a site, they're both the backbone of login and the source of whole classes of attacks (CSRF, session hijacking, XSS stealing cookies). That's why real cookies carry flags like HttpOnly (JavaScript can't read it — blunts XSS theft), Secure (only sent over HTTPS), and SameSite (not sent on cross-site requests — blunts CSRF). Our server already sets HttpOnly on its demo cookie as an early good habit. The full story is Phase 7 and Phase 24; the seed is planted here.

Part G — Keep-alive, and HTTP/1.1 vs /2 vs /3

Recall Phase 2's expensive ritual: before a single byte of data flows, TCP does its three-way handshake, costing a full network round-trip of pure latency. If every image, script, and click on a page opened a fresh TCP connection, you'd pay that handshake toll over and over. A single web page pulling 50 resources would mean 50 handshakes. That's brutal.

Keep-alive: reuse the connection

HTTP/1.1's fix is keep-alive (persistent connections): after answering a request, don't hang up — hold the same TCP connection open and let the client send its next request over it. Pay the handshake once, then reuse the pipe for request after request. This is precisely the "reuse the connection so you pay the handshake toll once" idea you predicted back in Phase 2's challenge, now a real, named feature.

The version ladder: /1.1 → /2 → /3

Keep-alive helped, but HTTP/1.1 still has a flaw: on one connection, requests are answered in order. If request 1 is slow, requests 2 and 3 wait behind it even though they're ready — a traffic jam called head-of-line blocking. The newer versions exist to kill it:

  • HTTP/1.1 (1997) — one request at a time per connection, answered in order. Keep-alive reuses the connection, but a slow response blocks the ones behind it. Browsers work around this by opening ~6 parallel connections per site. This is the version we implement.
  • HTTP/2 (2015) — multiplexing: many requests and responses interleave over a single connection at once, chopped into independent "streams," so a slow one no longer blocks the others. One lane became many logical lanes on the same road. Also compresses headers.
  • HTTP/3 (2022) — same multiplexing idea, but rebuilt on top of UDP instead of TCP (via a protocol called QUIC). Why? Because HTTP/2's streams still sat on one TCP connection, and TCP's own "deliver everything in order" rule reintroduced head-of-line blocking at the packet level. Moving to UDP lets each stream recover from a lost packet independently. It also folds the TLS handshake into the connection setup, so secure connections start faster.

You almost never implement HTTP/2 or /3 yourself — a web server (Nginx, Phase 9) or a CDN terminates them for you and often talks plain HTTP/1.1 to your app behind the scenes. So why learn the ladder? Because it's a perfect interview story about the same problem attacked at successive layers: latency of many requests → reuse the connection (keep-alive) → stop them blocking each other (multiplexing, HTTP/2) → stop the transport itself from blocking (UDP/QUIC, HTTP/3). Each version fixes the bottleneck the previous one exposed. Being able to tell that story — and say "and my hand-written server is the /1.1 rung" — shows you understand why the stack looks the way it does, not just its names.

Part H — HTTPS and the TLS handshake: how the padlock happens

Everything so far has been plain HTTP: text sent in the clear. Anyone who can see your packets on the way — the coffee-shop WiFi, your internet provider, a router in between — can read every byte, and worse, change it. For a link aggregator that might be survivable; for a password or a credit-card number it's a catastrophe. HTTPS is the fix, and the "S" is a single added layer called TLS (Transport Layer Security).

TLS wraps your ordinary HTTP in an encrypted tunnel and gives you three guarantees at once:

  • Confidentiality — eavesdroppers see only scrambled gibberish, not your data.
  • Integrity — if anyone tampers with the bytes in transit, both ends detect it and the connection breaks.
  • Authentication — you can be sure the server is really yourbank.com and not an impostor, because it presents a certificate vouched for by an authority your browser trusts.

It works through a TLS handshake that happens right after the TCP handshake, before any HTTP flows. Simplified to its essence:

The clever part is step 4: the two sides agree on a shared secret key without ever sending it across the wire in a form an eavesdropper could use, using public-key cryptography (the deep magic of Phase 7). After that, everything — the request line, the headers, the cookies, the body — is encrypted with that key. An eavesdropper sees only that you're talking to some server and roughly how much; the content is opaque.

📝 Where this lands for us

We do not implement TLS in our server — hand-rolling cryptography is a famously dangerous idea, and in the real world a layer in front (Nginx in Phase 9, or a cloud load balancer in Phase 20) "terminates TLS" and hands plain HTTP to your app behind a firewall. So your application code usually speaks plain HTTP and something else provides the padlock. But you should understand the handshake, because it explains the padlock icon, the "your connection is not private" warnings, why HTTPS adds latency, and why certificates expire and must be renewed. You'll set up real HTTPS in the Docker/Nginx and cloud phases.

The code we wrote

The Phase 2 server grew up. Same single file, project/server.py, but it now takes HTTP seriously. Here's the shape of what changed, and the key excerpts.

1. A real request parser

Phase 2 grabbed the first line and ignored the rest. Now parse_request splits the header block off at the blank line, pulls apart the request line into method/path/version, and folds every header into a dictionary with lower-cased names (so header lookups are case-insensitive, as the spec demands). Bad input raises BadRequest, which becomes a real 400 instead of a crash:

def parse_request(raw):
    # Headers are text; the body stays raw bytes. Split at the blank line.
    header_block, _, body = raw.partition(b"\r\n\r\n")
    lines = header_block.decode("iso-8859-1").split("\r\n")

    method, path, version = lines[0].split(" ")   # "GET /about HTTP/1.1"

    headers = {}
    for line in lines[1:]:
        name, value = line.split(":", 1)      # "Host: localhost:8000"
        headers[name.strip().lower()] = value.strip()   # fold name to lower-case
    return Request(method, path, version, headers, body)

2. A Response object with a full status-code table

Instead of gluing response strings by hand, a Response object holds a status code, a headers dict, and a body, and knows how to serialize itself to exact bytes. A REASON_PHRASES table maps every code we use to its human phrase. Crucially, to_bytes(include_body=False) is how we answer a HEAD request — identical headers (including the Content-Length the body would have had), but no body:

def to_bytes(self, include_body=True):
    reason = REASON_PHRASES.get(self.status, "Unknown")   # 200 -> "OK"
    self.headers["Content-Length"] = str(len(self.body))   # always the real byte count

    lines = [f"HTTP/1.1 {self.status} {reason}"]        # the status line
    for name, value in self.headers.items():
        lines.append(f"{name}: {value}")
    head = ("\r\n".join(lines) + "\r\n\r\n").encode("iso-8859-1")
    return head + self.body if include_body else head   # HEAD: headers only

3. Routes that each teach one HTTP idea

The route function is a plain if/elif chain (Phase 4's framework will replace it), but each branch demonstrates something concrete: a 301 redirect with a Location header at /old-home; a 401 gate that checks for an Authorization header at /secret; the cookie counter at /count; a POST /submit that appends a link (non-idempotent — POST again, get a duplicate); and PUT/DELETE on /links that are idempotent by contrast. Here's the idempotency pair, the heart of Part C made real:

# POST: each call ADDS a link -> send it twice, get two links (NOT idempotent)
if path == "/submit":
    LINKS.append({"title": title, "url": url, "points": 1})
    return Response(201, f"Added. Now {len(LINKS)} links.")   # 201 Created

# PUT: SET every link to 100 -> run 1x or 100x, same state (idempotent)
if path == "/links" and req.method == "PUT":
    for link in LINKS:
        link["points"] = 100
    return Response(200, "All links set to 100. Do it again — same result.")

4. Keep-alive: many requests over one connection

The biggest structural change. handle_connection is now a loop: it reads a request, answers it, and — if the client wants keep-alive and we're under our per-connection limit — keeps the socket open and reads the next request over the same connection. Because one recv can pull bytes belonging to the next request too, a per-connection buffer carries the spillover forward. The keep-alive decision honors both the client's wish and our own limits:

# Reuse this TCP connection only if the client wants it AND we're under our cap.
keep = req.wants_keep_alive() and served < KEEPALIVE_MAX_REQUESTS
resp.set("Connection", "keep-alive" if keep else "close")
conn.sendall(resp.to_bytes(include_body=(req.method != "HEAD")))
if not keep:
    return   # client asked to close, or we hit the limit — hang up
⚠️ The flaw is still here — on purpose

Keep-alive did not fix Phase 2's real limitation: the server still handles one connection at a time. In fact keep-alive slightly worsens it — now a single client can hold the one and only slot across many requests, and a slow client (run slow_client.py again) still freezes everyone. We are deliberately not solving concurrency yet. Feeling this limit sharpen is what will make threads, processes, and async event loops feel earned when we reach them.

Read the first line of each response (curl -i prints it): HTTP/1.1 301, then 401, then 200, then 404, then 405. On the 405, notice the Allow: GET, HEAD header — the server telling you which verbs that path does accept. On the first 401, notice WWW-Authenticate — the server telling you how to authenticate. The status codes from Part D, live.

Step 3 — cookies: the server "remembers" you

# Use a cookie jar file so curl stores and re-sends cookies, like a browser.
# Run this SAME line several times and watch the count climb:
curl -v -c jar.txt -b jar.txt localhost:8000/count

First run: the body says visited /count 1 time(s), and in the verbose output you'll see the server send Set-Cookie: visits=1. Run it again: the body says 2 time(s), because curl sent Cookie: visits=1 back (look for the > Cookie: line) and the server incremented it. You are watching a stateless server "remember" you purely through the cookie round-trip of Part F. (Delete jar.txt to start over at 1.)

Step 4 — POST vs PUT: idempotency you can count

# POST a new link. Run it TWICE and watch the link count go up BOTH times:
curl -i -X POST -d "title=Hello&url=http://example.com" localhost:8000/submit
curl -i -X POST -d "title=Hello&url=http://example.com" localhost:8000/submit

# See the duplicates you just created:
curl localhost:8000/links

# Now PUT twice — the SECOND run changes nothing further (idempotent):
curl -i -X PUT localhost:8000/links
curl -i -X PUT localhost:8000/links
curl localhost:8000/links   # every link is 100 points, once or twice, same result

What this proves: the two POSTs created two identical "Hello" links — repeating a non-idempotent call piled up. The two PUTs both set every link to 100 points — the second PUT left the world exactly as the first did. You just counted the difference between idempotent and not. This is Part C's whole point, made of real state you can list.

Step 5 — keep-alive: many requests, one connection

# curl reuses ONE connection for several URLs when you list them together:
curl -v localhost:8000/ localhost:8000/about localhost:8000/links 2>&1 | grep -iE "Re-using|Connected|GET |Connection:"

In terminal 1, watch the log: you'll see a single [CONN] conn#N accepted… followed by req#1, req#2, req#3 — three requests on the same connection — and each [SEND] line ending in keep-alive until the last. In terminal 2, curl prints Re-using existing connection for the 2nd and 3rd URLs. That's Part G working: one TCP handshake, many requests. Compare with curl -H "Connection: close" -v localhost:8000, where the server logs the response as closing and hangs up after one.

Step 6 — the flaw persists: stop the server

Optionally re-run Phase 2's three-terminal slow_client.py demo (terminal 1 server, terminal 2 python3 slow_client.py, terminal 3 curl localhost:8000) and confirm the fast curl still hangs until the slow client finishes — keep-alive didn't fix concurrency. Then, in terminal 1, press Ctrl+C:

Break it / observe it

Optional experiments. Each pokes one idea from this phase until you can see it move.

  1. Send raw HTTP by hand and read the status line. With the server running: printf 'GET /nope HTTP/1.1\r\nHost: x\r\nConnection: close\r\n\r\n' | nc localhost 8000 (install nc with sudo apt install -y netcat-openbsd). The very first line that comes back is HTTP/1.1 404 Not Found. You typed a request with your own hands and read the server's three-digit verdict — proof HTTP is just agreed-upon text.
  2. Trigger every family on purpose. Hit / (200), /old-home (301), /secret (401), /nope (404), -X POST /about (405). Use curl -s -o /dev/null -w "%{http_code}\n" localhost:8000/PATH to print just the number for each — a compact way to see the codes line up.
  3. Break the parser, get a 400. Send deliberately malformed HTTP: printf 'THIS IS NOT HTTP\r\n\r\n' | nc localhost 8000. Instead of crashing, the server answers HTTP/1.1 400 Bad Request and logs a [HTTP] bad request line. That's the BadRequest → 400 path: a server must never die on junk input.
  4. Watch a header decide behavior. Request /secret without and then with -H "Authorization: x". Same path, same method — the only difference is one header, and it flips a 401 into a 200. Headers aren't decoration; they steer the response.
  5. Prove HEAD sends no body. curl -si localhost:8000 (GET) shows headers and HTML; curl -sI localhost:8000 (HEAD) shows the same headers — including the same Content-Length — but stops there. The Content-Length advertises a body size that HEAD deliberately doesn't deliver.
  6. See keep-alive time out. Open a connection and just wait: nc localhost 8000, type one valid request and a blank line, read the reply, then don't send a second request. After ~5 seconds the server logs an [IDLE] line and closes the keep-alive connection. That's KEEPALIVE_TIMEOUT_SECONDS reclaiming an idle-but-open connection.

Glossary

New terms this phase, in plain words:

  • HTTP — HyperText Transfer Protocol: the request/response text protocol the web speaks over TCP. A request and a response are each just structured text: a first line, headers, a blank line, an optional body.
  • Request line / status line — the first line of a request (METHOD PATH VERSION) and of a response (VERSION CODE REASON).
  • Header — a Name: value metadata line. Names are case-insensitive. Carry identity, format, length, caching, cookies, connection control.
  • Body — the payload after the blank line; present on POST/PUT and on most responses. Its length is given by Content-Length.
  • Stateless — HTTP keeps no memory of you between requests; each arrives as if it were the first. Memory is added back deliberately with cookies.
  • Method / verb — the intent of a request: GET/HEAD (read), POST (create), PUT (replace), PATCH (partial update), DELETE (remove).
  • Safe — a method that only reads and changes nothing (GET, HEAD). Idempotent — a method where doing it once and doing it many times leave the same end state (GET, HEAD, PUT, DELETE), so retries are safe; POST is neither.
  • Status code — the response's three-digit verdict. Families: 1xx info, 2xx success, 3xx redirect, 4xx client error, 5xx server error. Know 200, 201, 301/302, 304, 400, 401, 403, 404, 405, 429, 500, 502, 503.
  • Cookie — text the server asks the browser to store (Set-Cookie) and the browser echoes back on every request (Cookie), letting a stateless server recognize a returning client.
  • Session vs JWT — two ways to use a login cookie: a random ID that points at server-side data (session; easy to revoke, needs a lookup) vs a signed self-contained token (JWT; scales without lookup, harder to revoke).
  • Keep-alive / persistent connection — reusing one TCP connection for many HTTP requests so the handshake is paid once, not per request.
  • Multiplexing — HTTP/2's interleaving of many requests over one connection so a slow one doesn't block the others; head-of-line blocking is the problem it solves.
  • HTTP/1.1 vs /2 vs /3 — one-at-a-time-in-order, multiplexed streams, and multiplexed-over-UDP (QUIC) respectively; each fixes the bottleneck the previous exposed.
  • HTTPS / TLS — HTTP wrapped in an encrypted, tamper-evident, authenticated tunnel. The TLS handshake exchanges hellos, verifies the server's certificate, and agrees a shared secret key before any HTTP flows.

Challenges & Questions

Totally optional, answers hidden — think first if you feel like it, or just reveal them.

Q1. A mobile app sends "create order" to your server, but the reply is lost to a flaky network. The app doesn't know if the order was created. Using this phase's ideas, why is "just retry" dangerous here, and what property would make it safe?

"Create order" is naturally a POST — and POST is not idempotent (Part C). So a blind retry might create a second order (the first request may have succeeded and only its reply got lost). The property that makes retries safe is idempotency: if the same call, repeated, leaves the same end state, retrying is harmless. Real systems make POSTs idempotent with an idempotency key — the client attaches a unique ID to the request; the server records "I already processed key ABC" and, on a retry with the same key, returns the original result instead of doing it again. That turns an unsafe retry into a safe one. (This is a real, heavily-used pattern — Stripe's API is a famous example — and it's Phase 21 material. You can already explain why it's needed.)

Q2. Someone builds a "delete this comment" feature as a normal link (<a href="/delete/42">). A week later, comments start vanishing on their own. What went wrong, in HTTP terms?

A clicked link is a GET, and GET must be safe — read-only (Part C). Hiding a destructive action behind a GET breaks that promise, and the web assumes GETs are safe: browsers pre-fetch links, "link preview" bots (in chat apps, email scanners) fetch URLs to generate thumbnails, and crawlers walk every link they find. Any of those quietly issued GET /delete/42 and deleted comments without a human ever clicking. The fix: destructive actions must use POST/DELETE, triggered by a form or JavaScript, never a bare link — and ideally protected against cross-site requests (Phase 7). "Never put a state change behind a GET" is one of the most important safety rules on the web.

Q3. Your monitoring shows a spike of 404s and, separately, a spike of 500s. Which should wake an engineer at 3am, and why? What would a spike of 401s suggest instead?

The 500s should wake someone — 5xx means the server failed (Part D): a bug, a crashed dependency, a database that's down. That's your fault and actively broken. A spike of 404s is 4xx — the client's side of the fence: usually harmless (a broken external link, a bot probing random URLs, a mistyped path), rarely an emergency, though a sudden 404 spike on pages that should exist can hint at a bad deploy that moved routes. A spike of 401s (unauthenticated) often means either an expired credential/config breaking legitimate clients, or someone probing your auth (credential-stuffing). This is exactly why real dashboards separate 4xx and 5xx rates — the leading digit routes the alarm to the right person. You now read that digit fluently.

Q4. A login cookie that literally contained user=admin would be a disaster. Explain the attack, then describe how BOTH the session approach and the JWT approach prevent it.

The attack is trivial: the browser stores the cookie as plain text and sends it on every request, so any user could just edit their cookie to user=admin and the server would believe them — instant impersonation. Both real designs prevent forgery, differently. Sessions: the cookie holds only a big random, meaningless ID (session=8f3a…); the mapping "this ID → user admin" lives on the server, where the client can't touch it. Guessing a valid random ID is infeasible, and the server can revoke it instantly by deleting the row. JWT: the cookie does contain user=admin, but with a cryptographic signature the server computes with a secret key. Edit the payload and the signature no longer matches, so the server rejects it; the client can't produce a valid signature without the secret. Session = "keep the truth on the server, hand out a claim-check"; JWT = "let the client hold the truth, but make it un-forgeable." Both are Phase 7, but the reasoning is pure Part F.

Q5. HTTP/1.1 keep-alive lets you reuse a connection, yet browsers still open about six connections per site, and HTTP/2 was invented anyway. What limitation of /1.1 forced both of those, and how does /2 remove it?

The limitation is head-of-line blocking (Part G): on a single HTTP/1.1 connection, requests are answered strictly in order, so one slow response stalls every request queued behind it, even though they're ready. Keep-alive removed the per-request handshake cost but not this ordering constraint. Browsers hacked around it by opening ~6 parallel connections per origin, so a stall on one didn't block all downloads — but 6 is a crude limit and each connection still has its own overhead. HTTP/2 removes the root cause with multiplexing: many requests and responses are split into independent streams and interleaved over a single connection, so a slow one no longer blocks the others and you don't need six connections. (And HTTP/3 goes further, moving onto UDP so a lost packet doesn't block unrelated streams — the same problem chased down one more layer.) The through-line: each version attacks the bottleneck the previous one left behind.

Q6 (stretch). Our server sets Content-Length honestly and refuses to trust a single recv. Imagine a server that instead just sent the body and closed the connection to signal "done," with no Content-Length. What breaks — and connect this to why keep-alive specifically needs Content-Length.

"Close the connection to signal done" actually works for a single response — that's literally how HTTP/1.0 and our Phase 2 server behaved (Connection: close). The problem appears the moment you want keep-alive: if the connection stays open for the next request, the client can no longer use "the connection closed" as the end-of-body marker — the connection is supposed to stay open. So it needs another way to know exactly where this response's body ends and the next response begins. Content-Length is that way: "the body is exactly N bytes; after those N bytes, the next response starts." Without it, keep-alive can't frame messages and the client would either hang waiting for more or run two responses together. (The alternative framing, for when you don't know the length up front — like streaming — is chunked transfer encoding, which sends the body in self-describing pieces; a natural next topic, and how our server would stream a large or generated response.) This is why our reader parses Content-Length and reads exactly that many body bytes before looking for the next request: message framing is what makes connection reuse possible at all.
