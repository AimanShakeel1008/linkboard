Why this phase exists

In Phase 1 we stayed inside a single computer: one CPU, its memory, its disk, its operating system. But LinkBoard is a website. A website's whole purpose is to let a computer you've never met — someone's phone on the other side of the planet — ask your computer for a page and get it back.

So we have to answer a question that sounds simple and absolutely is not: when you type an address and press enter, how do the right bytes travel to the right machine and come back?

The honest answer has a lot of moving parts, and this phase walks through every one of them from the bottom up. But here is the good news, and it's the same good news as last phase: none of it is magic. A web server is just a program that waits for bytes to arrive on a "socket," reads them, and writes bytes back. By the end of this phase you will have built that program yourself, from the rawest pieces the operating system offers, and watched real browser traffic flow through it in the logs.

Part A — The internet vs the web: roads vs trucks

People use "the internet" and "the web" as if they mean the same thing. They don't, and keeping them separate is the first step to understanding either.

Picture a country's road network: asphalt, intersections, traffic lights, on-ramps. The roads don't care what drives on them. Pizza-delivery scooters, moving vans, ambulances, tour buses — the road just carries whatever shows up.

The internet is that road network. It is the physical-and-logical system of cables, radio links, and routing machines that can carry any kind of data between any two connected computers. It's just the roads.

The web (short for World Wide Web) is one particular kind of traffic that drives on those roads: web pages, delivered using a specific set of rules called HTTP (which gets its own phase, Phase 3). The web is trucks of a certain brand. Other brands of truck use the same roads without ever touching the web: your email, a video call, a game's live updates, a file download, LinkBoard's future database traffic — all of these ride the internet, none of them are "the web."

📝 So what

This distinction is why we can build LinkBoard's server on raw sockets — the bare road — and only later add HTTP, the specific traffic rules. The road came first. The traffic rules are an agreement layered on top, and you can pick different agreements for different jobs.

What's a "protocol," then?

That word "agreement" is the whole idea of this phase, so let's name it. A protocol is a shared set of rules that two parties agree to follow so they can understand each other. That's it. It's not a piece of software or hardware — it's an agreement written down.

You already use protocols constantly. A phone call has one: you say "hello," the other person says "hello," you take turns, and "goodbye" ends it. If someone ignored that protocol — picked up and said nothing, or never let you speak — communication would fail even though the phone line worked perfectly. The line is the internet; the hello / take turns / goodbye is the protocol.

Everything in this phase is either a road (physical network) or a protocol (an agreement about how to use it): IP, TCP, UDP, DNS, HTTP — every one of those letters-soup names is the name of an agreement. Once you see them as "rules two computers promised to follow," the alphabet soup turns into a short, sensible list.

Part B — Packets & routers: your message becomes postcards

Why not just send the whole thing at once?

Say you want to send a friend a long letter — a hundred pages. You could mail it as one enormous bound book. But if that single package gets lost, or damaged, or stuck behind a bigger package, the whole letter is gone or delayed. And while your huge book hogs the mail truck, nobody else's mail fits.

The internet solves this the way you'd eventually think to solve it yourself: chop the message into small, equal-ish pieces and send each one separately. Each piece is called a packet. A packet is a small bundle of bytes — typically up to about 1,500 bytes — carrying a slice of your data plus a little header of bookkeeping (where it's going, where it came from, which slice number it is).

When you load LinkBoard's homepage, the page doesn't arrive as one blob. It's sliced into many packets, each finds its own way across the world, and your computer reassembles them back into the original page in the right order. This happens for every image, every click, every everything.

📝 Why slicing is genius
  • Sharing: many people's packets take turns on the same cable, interleaved — nobody's giant download blocks everyone else.
  • Resilience: if one packet is lost, you re-send just that one small packet, not the whole file.
  • Flexibility: different packets of the same message can travel different routes, whichever is clear at that instant.

Routers: the post offices of the internet

A packet has a destination written on it, but it has no idea how to get there — just like a postcard doesn't know the route, only the address. The devices that figure out the route are routers.

A router is a specialized computer that sits at a junction between networks. Its entire job: a packet arrives on one of its wires, the router looks at the destination address, consults a table of "to reach addresses like that, send them out this wire toward a router that's closer," and forwards the packet one hop onward. Then it forgets about it and handles the next packet.

No single router knows the whole path from you to LinkBoard's server. Each one only knows the next best step, like a chain of post offices each passing your postcard to the next office in roughly the right direction. A packet from your laptop might pass through ten to twenty routers on its way across the world, each one making a tiny "which way is closer?" decision. The astonishing part is that this decentralized, no-one's-in-charge, next-hop-at-a-time system reliably delivers trillions of packets a day.

🖥️ Try it now — watch the chain of post offices

There's a tool that shows you the actual routers your packets hop through to reach a destination. In WSL:

# Install it if needed (traceroute), then trace the path to a big site:
sudo apt install -y traceroute
traceroute google.com

Each numbered line is one hop — one router — with the time it took to reach it. You'll see the first hop is your own home router, then your internet provider's routers, then bigger backbone routers, then Google's. Some lines show * * * — that's a router choosing not to answer the probe (common and harmless). You are literally watching your postcard's route, office by office.

(Windows note: the same tool in plain Windows PowerShell is tracert google.com, if you'd rather see it from that side.)

Part C — IP addresses & ports: the street address and the apartment

The IP address: every machine's mailing address

For a packet to be delivered, the destination needs an address — a number every router understands. That number is an IP address (IP stands for Internet Protocol, the most fundamental agreement of all: "here is how we address and route packets").

The kind you've probably seen looks like this: 142.250.72.14. Four numbers, each from 0 to 255, separated by dots. This is IPv4 (version 4). Each of those four numbers is one byte (remember Phase 1 — one byte holds 0 to 255), so an IPv4 address is exactly four bytes = 32 bits.

Thirty-two bits gives about 4.3 billion possible addresses. In the 1980s that felt infinite. It isn't — there are more phones, laptops, servers, smart bulbs, and cars online than that. We ran out.

IPv6: the much bigger address book

The fix is IPv6, which uses 128 bits instead of 32. It looks alien the first time: 2001:0db8:85a3:0000:0000:8a2e:0370:7334 (eight groups of hexadecimal — Phase 1's base-16 notation, back again). How many addresses does 128 bits give? About 340 undecillion — a 39-digit number. Enough to give every grain of sand on Earth billions of addresses. We will not run out again. The world is slowly moving from IPv4 to IPv6; both run side by side today, which is why you see both forms.

🖥️ Try it now — find your own machine's addresses 
# Show this machine's network interfaces and their IP addresses:
ip addr

# Or just the IPv4 addresses, more compactly:
ip -4 addr show

Look for a line like inet 172.20.113.5/20. That 172.20.x.x (or 192.168.x.x, or 10.x.x.x) is a private address — it works only inside your home/WSL network, not on the public internet. Which raises a great question we answer in Part H: if your laptop's address is private, how does a reply from Google find its way back to you? (Answer: a trick called NAT.)

DHCP: who handed out that address?

You never typed your laptop's IP address in by hand. When a device joins a network, it shouts a request — "does anyone here assign addresses? I need one" — and a server (usually built into your home router) answers, "yes, here, use 192.168.1.37 for the next few hours, and by the way here's the router to send everything through." That automatic handout system is DHCP (Dynamic Host Configuration Protocol). It's why joining WiFi "just works" without you configuring anything. Yet another protocol — an agreement about how to ask for and grant addresses.

Ports: which program on the machine?

An IP address gets a packet to the right machine. But a single machine runs many programs that all use the network at once: a browser, an email client, LinkBoard's server, a music stream. When a packet arrives, which program gets it?

The answer is the port number — a second number, from 0 to 65535, that acts like an apartment number at a street address. The IP address is the building; the port is the specific apartment inside it. A full destination is written IP:port, for example 142.250.72.14:443.

Some port numbers are conventions everyone agrees on, so you don't have to be told them:

PortUsed forMeaning
80HTTPordinary (unencrypted) web traffic
443HTTPSencrypted web traffic (the padlock)
53DNSthe phone-book lookups from Part D
22SSHsecure remote login to another machine
5432PostgreSQLLinkBoard's future database (Phase 6)

These "well-known" conventions are why https://google.com works without you ever typing a port: your browser assumes port 443 for https. When we run LinkBoard's server this phase, we'll put it on port 8000 — a high, unreserved number, because ports below 1024 require admin rights to claim (a safety rule so random programs can't impersonate the mail service on port 25).

CS fundamental: addressing = which machine (IP) + which program (port). Almost every networked system you'll ever touch is reachable at some host:port.

Part D — DNS: the internet's phone book

The problem DNS solves

Machines find each other by IP address — cold numbers like 142.250.72.14. But no human wants to memorize that to visit a site, and worse, those numbers change: a company can move its servers and get new IPs next week. We need a layer of human-friendly names that can point at whatever the current numbers happen to be.

That layer is the Domain Name System, or DNS. It's the internet's phone book: you look up a name you can remember (linkboard.example.com) and it gives you back the current number (142.250.72.14) so your packets know where to go. The step of turning a name into an address is called resolving the name.

Why it's not one giant phone book

You might imagine one enormous file listing every website and its IP, copied to every computer. That could never work: there are hundreds of millions of domains, they change constantly, and a single list would be a single point of failure for the entire internet. So DNS is distributed and hierarchical — spread across many servers, organized like an address read right-to-left.

Take the name www.linkboard.example.com. Read it backwards, biggest zone first:

  1. The root (the invisible dot at the very end — the internet's "operator"). The root doesn't know your site, but it knows who handles each top-level ending.
  2. .com — the top-level domain (TLD). A set of servers responsible for all .com names. They don't know your specific site's address either, but they know who does: your domain's own name servers.
  3. example.com — the authoritative server. This is the server that actually holds the real answer for anything under example.com. It says: "www.linkboard.example.com? That's 142.250.72.14."

It's exactly like finding a person by "Country → City → Street → House." No single office holds every address; each level just points you one step closer. The clerk who does the asking-around on your behalf — walking from root to TLD to authoritative — is called a resolver, and it's usually run by your internet provider or a public one like 8.8.8.8.

The bit that makes it fast: caching & TTL

If every single click required that full root → TLD → authoritative walk, the web would crawl. So every answer comes with a TTL (Time To Live) — a number of seconds the answer may be reused before it must be looked up fresh. During that window, your resolver (and your own computer, and your browser) just remember the answer. Look up google.com once and the next thousand visits skip the whole walk and use the cached number. When the TTL expires, the cache forgets it, and the next lookup does the full walk again — which is exactly how a site can move to a new IP and have the world notice within, say, five minutes.

The simulation below lets you be the resolver: click through the chain once, then look the same name up again and watch the cache short-circuit the whole thing. That cache-hit-on-the-second-try is the entire point of TTL, and it's the same caching instinct from Phase 1 (keep popular answers close), now applied to names.

You are a DNS resolver. A program just asked you to resolve www.linkboard.example.com. Click Resolve the name and step through the chain. Then click it a second time and watch what the cache does. Use Advance the clock to expire the TTL and force a fresh walk.

This is a model of the mechanism, not a real resolver — real DNS has many more record types, retries, and layers of caching, and the walk happens in milliseconds. The real authority is dig +trace www.google.com in your own terminal, which shows the genuine root → TLD → authoritative walk against the live internet.

🖥️ Try it now — do a real DNS lookup, then watch the whole walk 
# Install the DNS tool if needed:
sudo apt install -y dnsutils

# 1. A plain lookup: name in, IP address(es) out.
dig google.com +short

# 2. The FULL walk — root, then TLD, then authoritative, printed step by step:
dig google.com +trace

# 3. See the TTL (the number of seconds before "look me up again"):
dig google.com

In the plain lookup you'll get one or more IP addresses. In +trace you'll see the query start at the root servers, get referred to the .com servers, and finally get the real answer from Google's own name servers — the simulation's three steps, but real. In the full dig google.com output, the ANSWER SECTION line has a number before the address (like 142.250.72.14 preceded by 212) — that number is the TTL in seconds. Run the command again a few seconds later and watch that number tick down: that's your resolver's cache counting toward the moment it must re-ask.

Part E — TCP vs UDP, and the handshake

We can now find a machine (IP) and the right program on it (port), and turn a name into an address (DNS). But packets, by themselves, are unreliable postcards: some get lost, some arrive out of order, some arrive twice. Who cleans that up? This is where two more protocols — TCP and UDP — split the world in two, and choosing between them is one of the most important design decisions in all of networking.

TCP: certified mail with a tracking number

TCP (Transmission Control Protocol) is the careful one. It takes the pile of unreliable packets and adds a layer of promises on top:

  • Reliable: every packet is numbered; the receiver acknowledges each one. If an acknowledgment doesn't come back, the sender re-sends. Nothing is silently lost.
  • Ordered: packets are reassembled in the original order, even if they arrived scrambled.
  • Connection-based: the two sides formally open a connection before sending data and close it when done — like a phone call with a clear "hello" and "goodbye."

The cost of all these promises is a little extra back-and-forth and a little delay. But for anything where correctness matters more than shaving milliseconds — loading a web page, sending an email, downloading a file, talking to a database — you want TCP. It's certified mail: slower, but it arrives complete, in order, guaranteed. LinkBoard uses TCP, and so does almost everything else you think of as "the web."

UDP: shouting across the room

UDP (User Datagram Protocol) is the opposite philosophy: fire the packets and don't look back. No connection setup, no numbering, no acknowledgments, no re-sends, no ordering. If a packet's lost, it's just gone.

Why would anyone want that? Speed and freshness. Think about a live video call. If one 20-millisecond sliver of audio gets lost, you do not want TCP to stop everything and re-send that stale sliver half a second later — by then the conversation has moved on. You'd rather drop it and keep up with the live moment. Live video, voice calls, online games, and (as it happens) DNS lookups all favor UDP: for them, a slightly-imperfect-but-current stream beats a perfect-but-late one. It's shouting across a room — fast, casual, and if one word is missed, oh well, keep going.

📝 The one-line rule of thumb

TCP when every byte must arrive correctly and in order (web, email, files, databases). UDP when being on-time beats being complete (live media, games, DNS). Neither is "better" — they're tools for different jobs, and the choice is a real system-design decision you'll make again and again.

The TCP three-way handshake

Because TCP is connection-based, before a single byte of your actual request can flow, the two computers must formally agree to open a connection. They do it with a three-step ritual called the three-way handshake. Using the phone-call analogy:

  1. SYN — your computer to the server: "I'd like to start a conversation. Can you hear me?" (SYN is short for "synchronize.")
  2. SYN-ACK — the server back to you: "Yes, I hear you, and I'd like to talk too. Can you hear me?" (One message doing double duty: acknowledging yours and starting its own.)
  3. ACK — your computer to the server: "Yes, I hear you. We're connected." ("ACK" = acknowledge.)

Three messages, and now a reliable, ordered connection exists in both directions. Only then does your browser send the actual GET / HTTP/1.1 request. This is also why a network round-trip has a real cost you can't avoid: even before asking for anything, you've spent one full round-trip just saying hello. Remember Phase 1's staircase — a network hop is "months" on the human scale — and now you know part of why: TCP politely shakes hands first.

Quick check: why does the handshake need three messages instead of two?

Because each side has to confirm that both directions of the connection work. Message 1 (SYN) lets the client say "can you hear me?" Message 2 (SYN-ACK) lets the server answer "yes, and can you hear me?" Message 3 (ACK) lets the client confirm "yes." After two messages only the server knows the link works in both directions; the third is what gives the client that same certainty. Two messages can't prove a two-way channel.

🖥️ Try it now — watch a real handshake

The -v ("verbose") flag on curl narrates the connection setup. In WSL:

curl -v http://example.com 2>&1 | head -20

Near the top you'll see a line like * Trying 93.184.216.34:80... — that's your computer picking the IP (from DNS) and reaching for port 80. The next * Connected to example.com line means the three-way handshake just completed successfully. Everything after that (> GET / HTTP/1.1) is the actual request — which only happened after the handshake. You're seeing the exact order this section described.

Part F — Bandwidth vs latency: the width of the pipe vs its length

People say a connection is "fast" or "slow" as if speed were one thing. It's two, and confusing them leads to real design mistakes. Picture water flowing through a pipe.

Bandwidth is how wide the pipe is — how much data can flow per second once it's flowing. It's measured in bits per second (Mbps, Gbps). More bandwidth = more data at once. This is what "100 Mbps internet" advertises. Bandwidth is what matters for big transfers: downloading a movie, uploading a video. A wider pipe moves the bulk faster.

Latency is how long the pipe is — how many milliseconds it takes for one bit to travel from one end to the other and (usually) for a reply to come back (a "round-trip"). Latency is set mostly by physical distance and the number of router hops. This is what matters for small, frequent, back-and-forth exchanges: a game input, a keystroke in a remote terminal, or — critically for us — each little request to a database.

📝 Why you can't fix latency with a bigger plan

Here's the trap. A server in Australia talking to a user in Germany has high latency no matter how much bandwidth either side buys, because the distance is fixed and light itself takes time to cross it (roughly 1 millisecond per 100 km, and that's the theoretical floor — reality is slower). Widening the pipe does nothing for a signal that simply has far to travel. This is why "just upgrade the connection" doesn't fix a laggy long-distance app, and why we put servers physically close to users (Phase 20's cloud regions and CDNs). It's also why making many tiny requests is so costly: each one pays the full latency toll again — the exact same "fixed per-request overhead" shape you met in Phase 1's disk race and syscall question, now wearing a network costume.

A concrete way to feel it: a transfer's total time is roughly (latency to get started) + (amount of data ÷ bandwidth). For a huge file, the second term dominates and bandwidth is king. For a tiny 200-byte database reply, the second term is a rounding error and latency is everything. Great system design is largely the art of not paying latency more times than you have to.

🖥️ Try it now — measure latency directly 
# ping sends tiny packets and times the round-trip — pure latency:
ping -c 5 google.com

# Compare a far-away target (times will be larger):
ping -c 5 example.com

Each line shows time=14.2 ms — that's the round-trip latency for one tiny packet. Notice the payload is only 64 bytes, so bandwidth is irrelevant here; you're measuring pure "length of the pipe." A nearby server might come back in single-digit milliseconds; one across an ocean in 150+ ms. Press Ctrl+C if you used a version without -c 5 and it won't stop.

Part G — The layer cake: why we stack all these protocols

We've now met a pile of protocols — IP, TCP, UDP, DNS, HTTP — and it might feel like a jumble. It isn't. They're organized into layers, stacked so that each layer does one job and trusts the layer below it to handle the details. This is one of the most powerful ideas in all of engineering, so it's worth seeing clearly.

The postal analogy, layer by layer

Think about mailing a birthday card. You do several jobs, each at its own "layer," and you deliberately don't think about the others:

  • You write the message ("Happy birthday!"). You care about the words, not the trucks.
  • You put it in an envelope with an address. Now it's addressed, but you don't care which trucks carry it.
  • The postal service routes it through sorting centers and trucks. They care about routing, not what the card says.
  • Roads and vehicles physically move it. They don't even know it's mail.

Each layer only talks to the one directly above and below it, through a clean handoff. The message-writer hands a sealed envelope to the post office; the post office hands bags to the trucks. Nobody needs to understand the whole system — and crucially, you can change one layer without touching the others. Switch from trucks to planes and the birthday message is unaffected. That independence is the entire payoff.

The internet's version

Networking uses this same stacked design. The formal teaching model is the OSI 7-layer model, but in practice the layers you'll actually name are these four, from top (closest to you) to bottom (closest to the wire):

LayerJobExamplesLinkBoard's version
Applicationthe actual conversation your program hasHTTP, DNS"GET / HTTP/1.1" — Phase 3
Transportreliable (or fast) delivery to the right programTCP, UDPTCP, via the socket we open
Internetaddressing & routing packets machine-to-machineIPIP addresses & routers, Parts B–C
Linkmoving bits over one physical hopWiFi, Ethernetyour WiFi to your router

When LinkBoard sends a page, the data goes down the stack on the sender's side (HTTP wraps it, TCP wraps that, IP wraps that, WiFi sends the bits) and up the stack on the receiver's side (WiFi receives bits, IP unwraps to find the machine, TCP reassembles in order, HTTP hands the page to the browser). Each layer on the receiving end reads only the envelope its counterpart wrote. It's envelopes inside envelopes inside envelopes — the technical name for this wrapping is encapsulation.

CS fundamental: layering / separation of concerns. Each layer solves one problem and hides its mess from the others, so any layer can be swapped without rewriting the rest. You'll see this same principle in software architecture (Phase 22), not just networks.
📝 Where the socket sits

The socket we're about to program is your code's doorway to the transport layer (TCP). You hand a socket some bytes; TCP, IP, and the WiFi driver — all living inside the kernel (Phase 1's superintendent) — handle every layer below. That's why we can build a "web server" without ever writing a single line about routers or radio waves: the layers below the socket are somebody else's solved problem. You get to stand on top of decades of it.

Part H — Firewalls, NAT & VPNs: gatekeepers and disguises

Three more pieces you'll bump into constantly. Each is easy once you have Parts A–G.

NAT: how a private address gets replies from the public internet

Back in Part C you found your machine has a private address (something like 192.168.1.37) that doesn't exist on the public internet. So when you request LinkBoard, how does the reply find its way back to your specific laptop, behind your router, sharing one public address with every other device in your home?

The trick is NAT (Network Address Translation), and your home router does it automatically. Think of your household having one public street address (the router's single public IP) while each family member has a private room number inside. When your laptop sends a packet out, the router rewrites the "from" address to its own public one, and writes down a note: "reply for this conversation → send back to laptop, room 37." When the reply arrives at the public address, the router checks its notes and forwards it to the right internal device. One public address, shared by a whole household of private ones, kept straight by the router's translation table. NAT is a big part of how the world limped along after IPv4 addresses ran out — many devices hiding behind one public address.

Firewalls: the bouncer with a guest list

A firewall is a filter that decides which network traffic is allowed through and which is blocked, based on rules. Picture a bouncer at a door with a guest list: "connections to port 443? Allowed. Connections to port 22 from outside the company? Denied." Firewalls run on individual machines (protecting one computer) and at network edges (protecting a whole network). They're why a fresh server in the cloud often refuses connections until you explicitly open the port you need — a wall that's closed by default is a safe wall. You'll meet this personally in Phase 9 and Phase 20, and probably curse it once before you love it.

VPNs: a private tunnel through public roads

A VPN (Virtual Private Network) makes two things happen at once. First, it builds an encrypted tunnel between your device and a VPN server: your traffic is scrambled so that anyone watching the public roads in between (a coffee-shop WiFi, your internet provider) sees only gibberish. Second, because your traffic exits to the wider internet from the VPN server, websites see the VPN server's address instead of yours — which is why a VPN can make you "appear" to be in another country. The roads are still the public internet; the VPN just wraps your trucks in an armored, address-swapping shell. It's a private conversation held in public — encryption (Phase 7's big topic) applied to the whole connection.

Part I — The whole journey: what happens when you press enter

Time to assemble every part into one story. You type http://linkboard.example.com and press enter. Here is the complete relay race, and every runner is something you now understand:

  1. Name → address (DNS, Part D). Your computer needs an IP, not a name. It asks its resolver; if the answer isn't cached, the resolver walks root → .com → authoritative and comes back with 142.250.72.14, plus a TTL saying how long to remember it.
  2. Open a connection (TCP handshake, Part E). Your computer sends a SYN to 142.250.72.14 on port 80; the server replies SYN-ACK; your computer sends ACK. A reliable, ordered connection now exists. This cost one round-trip (Part F's latency, spent before any real data).
  3. Ask for the page (HTTP — Phase 3). Over that connection your browser sends GET / HTTP/1.1 plus a few headers. This is the application layer riding on top of everything below.
  4. Down the stack, across the world, up the stack (Parts B, C, G). The request is wrapped by TCP, then IP, then your WiFi; it's chopped into packets; routers hop it toward the destination; the far machine unwraps the layers back up and hands the request to the server program listening on that port.
  5. The server does its job (this is the code we wrote!). LinkBoard's server — sitting in accept(), blocked, waiting — wakes up, reads the request bytes, decides which page the path asks for, and builds a response.
  6. The response comes back the same way — packets, routers, layers — and your browser reassembles the page and renders it (rendering is Phase 5).

Every phase from here builds on this skeleton. And the single step in bold — "the server wakes up, reads the request, builds a response" — is the thing we just wrote from scratch. Let's read it.

The code we wrote

Two files landed in project/ this phase — the first real LinkBoard code:

  • project/server.py — the raw-socket HTTP server.
  • project/slow_client.py — a cruel client that proves the server's deliberate flaw.

The five system calls that are a web server

Strip away everything and a TCP server is five kernel requests (Phase 1's syscalls — the doorbell), always in this order. Here's the heart of server.py:

# 1. socket(): ask the OS for a network endpoint (an unplugged phone).
#    AF_INET = IPv4 addresses; SOCK_STREAM = TCP (reliable, ordered).
server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)

# SO_REUSEADDR: let us re-grab the port immediately after a restart,
# instead of waiting out TCP's ~30s TIME_WAIT cool-down.
server_socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)

# 2. bind(): claim an address (all interfaces) and a port (8000).
server_socket.bind((HOST, PORT))

# 3. listen(): start queueing incoming connections (hold up to 16 waiting).
server_socket.listen(16)

while True:
    # 4. accept(): BLOCK until a client finishes the 3-way handshake,
    #    then return a NEW socket dedicated to that one client.
    conn, addr = server_socket.accept()
    handle_connection(conn, addr)   # read request, send response
    conn.close()                    # 5. close(): hang up.

That's the whole shape of every TCP server that has ever existed — Nginx, Postgres, your future LinkBoard at scale. Frameworks add convenience on top; this loop is always underneath. Notice accept() is exactly Phase 1's blocking I/O: the server sits there doing nothing, off the CPU entirely, until a connection wakes it.

Reading the request: bytes are a stream, not a message

The most important lesson hiding in the code is in how we read. A beginner expects "one recv() call gives me the whole request." TCP does not work that way. TCP hands you a stream of bytes with no built-in message boundaries — the request might arrive in one chunk or twenty, and you don't get to know in advance. So the server loops, collecting pieces, until it sees the blank line (\r\n\r\n) that HTTP uses to mark the end of the headers:

received = b""
while b"\r\n\r\n" not in received:      # until we see "end of headers"
    chunk = conn.recv(1024)             # BLOCKS; may return 1 byte or 1000
    if chunk == b"":                    # empty = the client hung up
        return
    received += chunk
    log("RECV", f"got chunk of {len(chunk)} bytes ...")

Every loop iteration logs the chunk size — which is how slow_client.py makes its point visible: it sends the request one byte at a time, so you'll see the server dutifully log [RECV] got chunk of 1 bytes over and over, stuck, unable to serve anyone else. That's the deliberate flaw, and it's the whole reason the slow client exists.

Building the response: it's just text with a strict shape

An HTTP response is plain text: a status line, some Label: value headers, one blank line, then the page. The build_response helper assembles it and — crucially — encodes it to bytes, because sockets carry only bytes, never Python strings (Phase 1, Part A: text is numbers plus an agreement, and UTF-8 is the agreement):

headers = (
    f"{status_line}\r\n"                            # "HTTP/1.1 200 OK"
    f"Content-Type: {content_type}; charset=utf-8\r\n"
    f"Content-Length: {len(body_bytes)}\r\n"        # how many body bytes follow
    f"Connection: close\r\n"                         # "I'll hang up after this"
    f"\r\n"                                           # the sacred blank line
)
return headers.encode("utf-8") + body_bytes

The Content-Length line matters more than it looks: it tells the client exactly how many body bytes to expect, so the client knows when the page is fully arrived instead of waiting forever for more. We'll parse requests and set status codes properly in Phase 3; for now the point is that a "response" is nothing but bytes in a shape both sides agreed on.

Step 2 — visit it

In terminal 2:

# The homepage, with full connection + header narration:

# The about page (plain text):

12:01:10.501 [RECV] #1 got chunk of 78 bytes (total so far: 78)
12:01:10.502 [HTTP] #1 parsed request line: method=GET path=/ version=HTTP/1.1 (read in 1 chunk(s))
12:01:10.502 [HEADER] #1 Host: localhost:8000
12:01:10.502 [HEADER] #1 User-Agent: curl/7.81.0

12:01:10.504 [SEND] #1 sent 612 bytes to 127.0.0.1:52344

What each line proves:

  • [CONN] accepted ... from 127.0.0.1:52344 — the three-way handshake (Part E) just completed; 127.0.0.1 is "this same machine" and 52344 is the random port your curl was given for its end.
  • [RECV] got chunk of 78 bytes — curl sent its whole request in one burst, so one recv got it all (contrast the slow client next).
  • [HTTP] parsed request line: method=GET path=/ — the server pulled the method and path out of the raw text. This is the application layer (Part G), by hand.
  • [ROUTE] / -> homepage — the server decided which page the path earns.
  • [SEND] sent 612 bytes then [WAIT] again — response delivered, connection closed, back to sleep. One full cycle.

In terminal 2, curl -v shows you the other side of that same conversation: * Connected to localhost (handshake done), > GET / HTTP/1.1 (its request), < HTTP/1.1 200 OK (our response), then the HTML. You are watching both ends of one TCP connection at once.

Try the browser too: open http://localhost:8000. You'll see the styled list of links — and in terminal 1, often several connections in a row, because browsers fetch extras (like a favicon) and open spare connections. Some may show up as [TIMEOUT] lines — that's a browser opening a connection "just in case" and never speaking; our 5-second timeout hangs up on it. That's normal and the log says so.

Step 3 — the main event: one slow client freezes everyone

This is the payoff of the whole phase — Phase 1's blocking I/O, now hurting real users. You need three terminals. Terminal 1 still runs the server.

In terminal 2, unleash the slow client:

python3 slow_client.py

It connects and starts dribbling its request one byte every 0.25 seconds, printing as it goes. In terminal 1 you'll see the server stuck receiving single bytes:

12:05:00.000 [CONN] #5 accepted connection from 127.0.0.1:52360
12:05:00.001 [RECV] #5 got chunk of 1 bytes (total so far: 1)
12:05:00.251 [RECV] #5 got chunk of 1 bytes (total so far: 2)
12:05:00.502 [RECV] #5 got chunk of 1 bytes (total so far: 3)
   ... one byte at a time, for ~14 seconds ...

Now, while that's still dribbling, go to terminal 3 and make a perfectly normal, fast request:

curl localhost:8000

Watch what happens: the curl in terminal 3 HANGS. It sits there, getting no response, no logs about it in terminal 1 — nothing — until the slow client in terminal 2 completely finishes. Only then does terminal 1 print [CONN] #6 and serve terminal 3's request, and the hung curl suddenly completes.

What this proves — and why it's the whole point: our server handles exactly one connection at a time. It's stuck in recv() waiting on the slow client (blocking I/O, Part H of Phase 1), so it literally cannot reach the next accept() to notice terminal 3 knocking. One slow visitor froze the entire shop for everyone. This is not a bug in our code — it's the fundamental limitation of the single-connection design, and feeling it here is what makes every later scaling technique (threads, async, multiple processes, load balancers) obviously necessary instead of arbitrary. We keep the flaw this phase on purpose; we earn the fixes later.

Quick check: is the frozen server a bug you should fix by writing more careful code?

No — and that's the whole point. The freeze is the unavoidable consequence of a design choice: one connection handled start-to-finish before the next is accepted. No amount of tidier code removes it; only a different architecture does (a thread/process per connection, or a non-blocking event loop). Recognising "this is a design limit, not a coding mistake" is exactly the judgement that separates memorised patterns from real system-design reasoning.

You've just walked into one of the most-asked systems-design interview arcs, from the ground floor. "Here's a server that handles one connection at a time — how do you make it serve thousands?" is a question you can now answer from mechanism, not memory: thread-per-connection (simple, memory-heavy), process-per-connection (isolated, heavier), or a single-threaded event loop (scales furthest, hardest to reason about) — and in production, usually a blend (a pool of processes each running an event loop, behind a load balancer). Every one of those is a later phase of this course. Keeping the naive server in front of you is what makes each fix feel earned rather than cargo-culted.

^C
12:10:00.000 [SHUTDOWN] Ctrl+C received — closing the listening socket. Bye.

Break it / observe it

Optional experiments. Each one pokes a concept from this phase until you can see it move.

  1. Prove ports are apartments. With the server running on 8000, try curl localhost:8001 (a port nobody's listening on). You get Connection refused instantly — the machine is there (the building exists) but no program is home in that apartment. Change PORT = 8000 to PORT = 9000 in server.py, restart, and now curl localhost:9000 works while 8000 refuses. The address found the building; the port chose the room.
  2. See the stream have no message boundaries. In slow_client.py, lower DELAY_BETWEEN_CHARS to 0.01 and re-run. Now the server's [RECV] logs show chunks of several bytes at once instead of exactly 1 — the same request, "clumped" differently by TCP purely based on timing. Nothing in your code changed; TCP just handed the bytes over in different-sized handfuls. That's why we loop until \r\n\r\n instead of trusting one recv.
  3. Watch DNS caching with your own eyes. Run dig google.com twice in a row, quickly. Compare the TTL number in the ANSWER SECTION — the second run's is smaller. That number is your resolver's cache counting down; you're watching the exact mechanism the simulation modeled. Wait a minute and run again — it dropped further, or reset (a fresh walk) if it hit zero.
  4. Feel latency vs bandwidth. ping -c 3 localhost (times near 0.0 ms — the pipe has no length, it's the same machine) versus ping -c 3 google.com (real milliseconds — real distance). Same tiny packets, wildly different latency, and bandwidth had nothing to do with either. Distance is the variable.
  5. Send raw HTTP by hand, no curl. With the server running: printf 'GET /about HTTP/1.1\r\nHost: x\r\n\r\n' | nc localhost 8000 (install with sudo apt install -y netcat-openbsd if needed). nc ("netcat") just opens a TCP connection and pipes your bytes in raw. You typed HTTP with your own hands and got a real response — proving HTTP is just agreed-upon text over a socket, nothing more. (If nc hangs after the reply, that's it waiting for the server to close — Ctrl+C out.)
  6. Make the timeout fire on purpose. nc localhost 8000 and then just... don't type anything. After 5 seconds the server logs a [TIMEOUT] line and hangs up on you, and nc exits. That's the CLIENT_TIMEOUT_SECONDS guard protecting the server from a silent connection holding a slot forever — a small taste of why real servers are paranoid about slow/idle clients.

Glossary

New terms this phase, in plain words:

  • Internet vs web — the physical/logical network that carries any data (the roads) vs one kind of traffic on it, web pages over HTTP (certain trucks).
  • Protocol — a shared set of rules two parties follow so they can understand each other. Not software; an agreement.
  • Packet — a small bundle of bytes (≈ up to 1,500) that a message is chopped into for sending; each carries a header with source, destination, and sequence info.
  • Router — a machine at a network junction that reads a packet's destination and forwards it one hop closer, knowing only the next best step.
  • IP address — a machine's numeric mailing address. IPv4 = 32 bits (like 142.250.72.14); IPv6 = 128 bits (like 2001:db8::7334), created because IPv4 ran out.
  • Private vs public IP — an address that works only inside a local network (e.g. 192.168.x.x) vs one reachable across the whole internet.
  • DHCP — the system that automatically hands a device an IP address (and its router) when it joins a network.
  • Port — a number (0–65535) identifying which program on a machine a packet is for; the apartment number to the IP's building. Well-known: 80 (HTTP), 443 (HTTPS), 53 (DNS), 22 (SSH).
  • DNS — the internet's phone book: turns a human name into a machine IP. Resolver — the clerk that does the lookup for you. Root / TLD / authoritative — the three levels of the lookup hierarchy (operator → .com handler → the domain's own answer-holder). TTL — how many seconds an answer may be cached before re-looking-up.
  • TCP — reliable, ordered, connection-based transport (certified mail). UDP — fast, connectionless, no guarantees (shouting across a room). Three-way handshake — the SYN / SYN-ACK / ACK ritual that opens every TCP connection.
  • Bandwidth vs latency — how much data per second the pipe carries (its width) vs how long one bit takes to travel and return (its length). Bandwidth rules big transfers; latency rules small frequent ones and can't be bought away.
  • OSI / layer model — the design that stacks protocols so each layer does one job and trusts the one below. Practical four: Application (HTTP), Transport (TCP/UDP), Internet (IP), Link (WiFi). Encapsulation — each layer wrapping the one above in its own envelope.
  • Socket — your program's endpoint for network communication; the doorway to the transport layer the kernel manages for you.
  • NAT — Network Address Translation: a router letting many private addresses share one public address by rewriting and tracking connections.
  • Firewall — a filter that allows or blocks traffic by rule (the bouncer with a guest list).
  • VPN — an encrypted tunnel that hides your traffic from onlookers and makes it exit from the VPN server's address.

Challenges & Questions

Totally optional, answers hidden — think first if you feel like it, or just reveal them.

Q1. You unplug your router and plug it back in. For a few seconds, typing google.com fails, but pinging 8.8.8.8 (a raw IP) works immediately. Using this phase's parts, what's the most likely explanation?

Your machine reached the internet (IP-level connectivity is back — the raw IP ping proves routing works), but it can't yet turn names into addresses because its DNS resolver isn't reachable or configured yet — DHCP (Part C) hadn't finished handing your machine its resolver address after the reconnect. Names need DNS (Part D); raw IPs skip that whole step. Give DHCP a moment to finish and google.com starts working. This is a classic "the internet is up but DNS is down" situation, and now you can diagnose it precisely.

Q2. A video-call app and a file-download app both run "over the internet." One should use TCP, the other UDP. Which is which, and what exactly would go wrong if you swapped them?

The file download wants TCP: every byte must arrive correctly and in order, or the file is corrupt — a slightly-late-but-perfect file is exactly right. The video call wants UDP: being on-time beats being complete, so a lost sliver of audio should be dropped, not re-sent stale. Swap them and both break in telling ways. A download over UDP could silently corrupt (missing/reordered chunks, no re-sends). A video call over TCP would stutter and lag: when a packet is lost, TCP halts the stream to re-send that now-useless old audio, so the call falls further and further behind live. The mismatch isn't "slower" — it's the wrong guarantee for the job (Part E).

Q3. Our server logs [RECV] got chunk of 1 bytes many times for the slow client, but usually just one [RECV] for curl. Since both send the identical request, why the difference — and why does the server loop instead of calling recv() once?

TCP delivers a stream of bytes with no message boundaries (Part E, and the code walkthrough). How the stream gets split into recv-sized handfuls depends purely on timing: curl sends its whole request in one burst, so it's usually sitting in the kernel's buffer as one lump when we recv; the slow client feeds one byte every 0.25s, so each recv catches exactly one. Neither is "more correct" — TCP guarantees the bytes and their order, not their grouping. That's precisely why the server must loop until it sees the end-of-headers marker (\r\n\r\n) rather than trusting a single recv to hand over a whole request. Assuming "one recv = one message" is one of the most common networking bugs there is.

Q4. Someone says: "DNS caching is dangerous — if a site moves to a new server, everyone with the old address cached will hit the dead one." Using TTL, explain why this is a managed, deliberate tradeoff rather than a flaw.

Every cached answer carries a TTL — a countdown after which it's discarded and re-looked-up (Part D). So a stale answer only lives for at most the TTL, then the world re-resolves and picks up the new address. Operators choose the TTL to trade speed against agility: a long TTL (hours) means fast repeat lookups but slow propagation when you move; a short TTL (a minute) means slightly more lookup traffic but near-instant propagation. Before a planned move, operators lower the TTL in advance so caches expire quickly during the switch, then raise it again afterward. It's a deliberate dial between "cheap and fast" and "nimble," not an accident — the same speed-vs-freshness tension you'll meet in every cache in this course (Redis, Phase 8).

Q5. The three-way handshake costs one full round-trip before any data. LinkBoard will eventually make many requests to the same server. Using Part F's latency idea, guess a strategy real systems use to avoid paying that handshake toll over and over. (Both real answers are hinted at across this phase.)

1. Reuse the connection. Instead of a fresh handshake per request, keep the TCP connection open and send many requests over it — you pay the handshake round-trip once, then amortize it across everything that follows. This is exactly what HTTP "keep-alive" and connection pooling do (Phase 3 for HTTP, Phase 6 for database connection pools — where opening a connection per query is shown to be a disaster). 2. Put the server physically closer so each round-trip is cheaper in the first place — fewer kilometers, less latency (Part F; this becomes cloud regions and CDNs in Phase 20). Both are the same insight from a different Phase-1 costume: don't pay a fixed per-request cost more times than you have to.

Q6 (stretch). Our server's flaw is that one slow client blocks all others. Before reading later phases, brainstorm three fundamentally different ways you could fix it — and note which Phase-1 idea each one leans on.

All three are real, and each is a future phase: (a) One thread per connection — hand each accepted connection to its own thread so a slow one blocks only itself, not the accept loop. Leans on Phase 1's threads (and inherits their shared-memory dangers). (b) One process per connection — same idea with processes instead of threads: safer isolation, heavier cost. Phase 1's processes. (c) One thread, but never block — ask the kernel "which of my many connections has data ready right now?" and only touch those (an "event loop" / async model). This leans hardest on Phase 1's blocking-vs-non-blocking I/O: instead of blocking on one client, you juggle thousands, working only on whichever is ready, like one waiter watching every table at once. Real servers mix these (a pool of processes, each running an event loop). You built the naive version; now you can appreciate exactly what each fix buys and costs. That appreciation — not the code — is the point.
